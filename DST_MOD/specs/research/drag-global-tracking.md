## ground

**1.** 
  - **summary:** Vanilla DST implements robust, focus-independent drag tracking with TWO canonical mechanisms, both visible in /tmp/dstscripts/scripts (Windows path: C:\Users\Marcos\AppData\Local\Temp\dstscripts\scripts).

MECHANISM 1 — the global mouse-move subscription (the truly "track-globally for the whole drag" answer). `TheInput:AddMoveHandler(fn)` registers a callback on the global `move` event that fires on EVERY frame the mouse moves, with NO dependency on focus/hover or on the cursor staying over any widget. It returns an EventHandler object; you store it and call `:Remove()` to unsubscribe (on mouseup). The canonical base-class API built on this is `Widget:FollowMouse()` / `Widget:StopFollowMouse()`. The handler signature is `function(x, y)` and `(x,y)` are absolute screen coords; `TheInput:GetScreenPosition()` (returns Vector3 from `TheSim:GetPosition()`) gives the current cursor at any time, e.g. to seed the start position. This is the mousedown→AddMoveHandler→(track every move regardless of hover)→mouseup→handler:Remove() pattern, and it is exactly NOT tied to the widget staying under the cursor.

MECHANISM 2 — focus-lock (what the scrollbar thumb drag actually uses). On `SetOnDown`, call `TheFrontEnd:LockFocus(true)` and override the widget's `OnLoseFocus` to a no-op; on up/click call `TheFrontEnd:LockFocus(false)`. `FrontEnd:DoHoverFocusUpdate` is what reassigns focus to whatever entity is under the mouse each frame, and it is gated by `if self.tracking_mouse and not self.focus_locked`. So locking focus freezes it on the dragged widget; its `SetWhileDown`/`OnUpdate` keeps firing even as the cursor leaves the bar. During drag it reads the always-current global cursor via `TheFrontEnd.lastx`/`TheFrontEnd.lasty` (updated in `FrontEnd:OnMouseMove`), NOT via the widget's own hover.

WHY THE REPORTED BUG HAPPENS (root cause in button.lua): `Button:OnUpdate` only calls `whiledown()` while `self.down` is true; `Button:OnControl` early-returns `if not self.focus`; and `Button:OnLoseFocus` sets `self.down = false`. So a fast move that takes the cursor off the title-bar ImageButton drops its focus, clears `down`, and the SetWhileDown/update loop stops — precisely the "panel STOPS when you move fast" symptom. Both vanilla mechanisms solve this; only Mechanism 1 (AddMoveHandler) is genuinely independent of focus.

**2.** 
  - **summary:** For a global mouse-tracked drag in DSTP's draggable panel, the correct primitive is TheInput:AddMoveHandler(fn) (input.lua:135-137). It subscribes to the Input.position EventProcessor's "move" event, which is fired engine-side every time the OS reports a cursor move via OnPosition->Input:UpdatePosition->self.position:HandleEvent("move", x, y) (input.lua:746-748, 155-159). This dispatch is global and completely independent of widget focus/hover: EventProcessor:HandleEvent just iterates every registered handler (events.lua:42-49) with no focus check anywhere. So it keeps firing no matter how fast the cursor moves or whether it has left the title-bar widget — exactly what a drag needs.

Two of the methods named in the task DO NOT EXIST in this codebase: there is no Input:AddMouseMoveHandler and no Input:AddPositionHandler (grep of input.lua, frontend.lua, and the whole scripts dir returns nothing). The real names are AddMoveHandler (position/move), AddMouseButtonHandler (button down/up only, not movement), AddControlHandler / AddGeneralControlHandler (control events), AddMouseMoveHandler is just a misremembering of AddMoveHandler.

Unsubscribe: AddMoveHandler (and all Add*Handler) return an EventHandler object; call handler:Remove() (events.lua:7-9, which calls processor:RemoveHandler(self), events.lua:29-36). Store the handle on mousedown, drop it on mouseup. (Internally Input:AddMoveHandler returns self.position:AddEventHandler("move", fn), input.lua:135-137 -> events.lua:17-27, which returns the handler.)

On the OnUpdate/StartUpdating path: Widget:StartUpdating() (widget.lua:485-487) just registers the widget into TheFrontEnd.updating_widgets, and FrontEnd:Update drives them at frontend.lua:876-886. That loop is gated ONLY by k.enabled (frontend.lua:881-883: "if k.enabled then k:OnUpdate(dt) end") — there is NO focus check. So OnUpdate is NOT focus-gated: a StartUpdating widget keeps getting OnUpdate every frame even after the cursor leaves it and it loses focus. This contrasts with Widget:OnControl, which IS focus-gated ("if not self.focus then return false end", widget.lua:127).

Conclusion on the bug: OnUpdate does NOT stop when the bar loses focus — so the "panel stops on fast move" symptom is NOT OnUpdate dying. The real cause is the cursor-delta source. If the move is computed from a per-widget hover/focus signal or polled position that only updates while hovered, fast moves outrun it. The robust fix: on mousedown, AddMoveHandler(fn) and in fn read the absolute cursor each event (TheInput:GetScreenPosition() / TheSim:GetPosition(), input.lua:246-248, 293-294) to compute delta against the last sample, move the panel, and on mouseup call handler:Remove(). This tracks globally regardless of focus and also kills the startup delay (no waiting for the first OnUpdate tick). You can keep StartUpdating purely for rendering if desired, but the position source should be the move handler, not focus-dependent state.

## diag

- **rootCause:** Two distinct symptoms, both stemming from the drag being driven by the title-bar widget's own focus/hover state instead of a global input subscription.

(1) STOPS ON FAST MOVE — the committed/buggy MakeDraggable (commit e377cea, ui_widgets.lua) ran the drag in `dragArea.OnUpdate` guarded by `if not TheInput:IsControlPressed(CONTROL_PRIMARY) then dragging=false; dragArea:StopUpdating(); return`. On the HUD, mouse-button control state is resolved through FOCUS-based routing: only the focused/hovered widget sees CONTROL_PRIMARY as held (the same focus-based HUD click routing this file documents at lines 327-330; button.lua:59). A fast flick outruns the invisible title-bar ImageButton, the cursor exits its hit region, dragArea loses focus/hover, so IsControlPressed(CONTROL_PRIMARY) returns false even though the physical LMB is still held. The guard then LATCHES dragging=false and calls StopUpdating(), de-registering the update loop — the drag dies permanently and never resumes even if the cursor returns. Note: it is NOT OnUpdate being focus-gated (StartUpdating registers the widget in the screen's updating_widgets list, which FrontEnd:Update drives every frame gated only by k.enabled, no focus check). The killer is the focus-coupled IsControlPressed guard self-terminating via StopUpdating().

(2) STARTUP DELAY — a real ~1-frame latency, two causes: (a) OnMouseButton(down) called dragArea:StartUpdating(), but StartUpdating only schedules the widget into the update list, so the first OnUpdate runs on the NEXT frame, not the same frame; (b) on that first OnUpdate frame, lastx/lasty had just been seeded to the current cursor, so dx/dy ≈ 0 and the `if dx~=0 or dy~=0` move is skipped — visible motion only begins the frame AFTER the first cursor movement.
- **whyStopsOnFastMove:** The drag's continuation was conditioned on `TheInput:IsControlPressed(CONTROL_PRIMARY)` evaluated inside the title-bar widget's OnUpdate. CONTROL_PRIMARY (held LMB) on the HUD is reported per the FOCUSED widget's control routing, not globally. A fast cursor flick leaves the invisible title-bar ImageButton's hit region before focus can follow; dragArea is no longer the focus/hover target, so IsControlPressed(CONTROL_PRIMARY) returns false. The guard interprets that false as 'mouse released off-area', sets dragging=false and calls dragArea:StopUpdating(), which removes the widget from the screen's update loop. Because both the flag and the de-registration are latched, the drag is permanently dead — re-entering the bar does nothing. The per-frame delta math itself is fine (it re-reads the absolute global cursor each frame, no integration drift), so a fast move would have tracked correctly if the focus-coupled guard hadn't already killed the loop. The cure is to stop deriving 'still dragging' from any focus-dependent signal and instead drive both the per-move tracking and the end-of-drag off global input subscriptions that fire regardless of focus.
- **theFix:** Replace the OnUpdate + IsControlPressed loop with two GLOBAL input subscriptions registered on mousedown and removed on mouseup, so tracking is independent of whether the cursor stays over the title bar. This is exactly what the working tree already contains (uncommitted) at ui_widgets.lua MakeDraggable (lines 381-429) and it is the correct fix:

- On mousedown (dragArea.OnMouseButton, button==LEFT, down==true): seed lastx,lasty from TheInput:GetScreenPosition(), MoveToFront, then move_handle = TheInput:AddMoveHandler(fn). AddMoveHandler (input.lua:135 -> position:AddEventHandler("move", fn)) fires on EVERY OS-reported cursor move via Input:UpdatePosition->position:HandleEvent("move",x,y); EventProcessor:HandleEvent iterates all handlers with NO focus check (events.lua), so it keeps firing no matter how fast you move or whether the cursor left the bar. This also kills the startup delay (first move event fires immediately, no waiting for the next-frame OnUpdate tick).
- The move fn computes dx,dy = mx-lastx, my-lasty against the global absolute cursor, updates lastx,lasty, and applies the delta to target:GetPosition() converted by the parent scale (divide by parent scale — see below).
- End-of-drag is handled by a SECOND global subscription: btn_handle = TheInput:AddMouseButtonHandler(fn) (input.lua:131), whose fn ends the drag when btn==LEFT and down==false. This guarantees the drag ends even if LMB is released OFF the bar. Also handle the local dragArea.OnMouseButton up branch (release while still over the bar) by calling the same stop().
- stop(): both handles are EventHandler objects returned by Add*Handler; call move_handle:Remove() and btn_handle:Remove() (events.lua:7-9) and nil them. Guard re-entry with `if move_handle then return true end`.

PARENTSCALE 1.5: AddMoveHandler delivers ABSOLUTE SCREEN-PIXEL deltas, but `target` lives in its parent's LOCAL space, and the parent is scaled by 1.5. A screen delta of dx pixels equals a local delta of dx/1.5. So DIVIDE the pixel delta by the parent scale before applying: target:SetPosition(tp.x + dx/sx, tp.y + dy/sy) where sx,sy come from target.parent:GetScale(). (The old buggy code applied the raw dx with no division, which also made the panel move 1.5x too far under the cursor — the fix corrects that too.) Read scale defensively each move (parent may be nil; fall back to 1,1; guard against a 0 scale component).
- **fixPseudocode:** local function MakeDraggable(dragArea, target)
    local lastx, lasty = 0, 0
    local move_handle, btn_handle = nil, nil

    -- screen px -> target local space: parent is scaled (e.g. 1.5), so divide.
    local function scaleXY()
        local parent = target.parent
        if parent and parent.GetScale then
            local s = parent:GetScale()
            if s then return (s.x ~= 0 and s.x) or 1, (s.y ~= 0 and s.y) or 1 end
        end
        return 1, 1
    end

    local function stop()
        if move_handle then move_handle:Remove(); move_handle = nil end
        if btn_handle  then btn_handle:Remove();  btn_handle  = nil end
    end

    dragArea.OnMouseButton = function(self, button, down, x, y)
        if button ~= _G.MOUSEBUTTON_LEFT then return false end
        if down then
            if move_handle then return true end          -- already dragging
            local p = _G.TheInput:GetScreenPosition()    -- absolute global cursor
            lastx, lasty = p.x, p.y
            target:MoveToFront()
            -- GLOBAL move: fires every cursor move, no focus/hover dependency.
            move_handle = _G.TheInput:AddMoveHandler(function(mx, my)
                local dx, dy = mx - lastx, my - lasty
                lastx, lasty = mx, my
                if dx ~= 0 or dy ~= 0 then
                    local sx, sy = scaleXY()
                    local tp = target:GetPosition()
                    target:SetPosition(tp.x + dx / sx, tp.y + dy / sy)  -- /1.5
                end
            end)
            -- GLOBAL mouse-up: ends drag even if released off the bar.
            btn_handle = _G.TheInput:AddMouseButtonHandler(function(btn, bdown)
                if btn == _G.MOUSEBUTTON_LEFT and not bdown then stop() end
            end)
        else
            stop()   -- released while still over the bar
        end
        return true
    end
end
-- NOTE: no OnUpdate, no StartUpdating/StopUpdating, no IsControlPressed.
