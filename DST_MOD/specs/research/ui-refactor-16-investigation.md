## ground

**1.** 
  - **summary:** ui_widgets.lua has two parallel rendering layers. LAYER A (legacy, lines ~104-429) is five hardcoded builders (CreateNotification/CreateLabel/CreatePanel/CreateButton/CreateProgressBar) each with an Update* partner, dispatched via CREATORS/UPDATERS tables keyed by widget `type`. LAYER B (generic tree renderer, lines ~431-795) builds a recursive widget tree from `{type, ...props, children}` nodes (col/row/tabs/text/icon/image/button/bar/spacer/panel), with addressable in-place prop patching (Register/SetProps) and a generic clickable wiring (MaybeClickable via OnControl/SetClickable). The tree renderer can already express most of what the legacy panel/button/bar/label builders do, but has a few gaps that block deleting the legacy layer: notification slide-in tween + auto-dismiss timer, label's 11-position anchor placement on individual primitives, per-text word-wrap region on legacy label, and the legacy single-widget Update* path that the backend `update` action targets by `type`. Both layers share the same action surface (create/update/destroy/destroy_group/set_text/set/ui_set/clear) in ProcessCommand; the `tree` type is itself registered in the legacy CREATORS/UPDATERS tables, so the two layers are bridged rather than cleanly separated. The "batch" envelope is NOT handled here anymore (fan-out moved to modmain's dstp_ui_dirty router, per comment lines 1089-1093).

**4.** 
  - **summary:** Traced the ui_callback contract end-to-end across the DST Lua mod and the FluxStack backend. A click anywhere in the UI funnels through ONE callback function (UIWidgets._callback_fn / ctx.callback_fn), which sends MOD_RPC UICallback to the server, where modmain's UICallback handler turns it into a `ui_callback` event via PushEvent. That event rides the normal /dst/sync POST to the backend, where FlowEngine.evaluateEvent matches it to any `trigger` node whose event_type is `ui_callback`, exposing the click as `{{trigger.callback}}` / `{{trigger.widget_id}}`. A new clickable text/icon/image must NOT invent a new RPC — it must carry a `callback` string on its tree node and route the click through the SAME `ctx.callback_fn(cb, ctx.root_id)` already used by `button`, exactly as the generic `MaybeClickable` helper does (which is what makes any Text/Image clickable today).
  - **facts:** 
    - (1) FULL PATH — click → callback → ui_callback event:
[a] Top-level widget click (the dedicated `button` widget): E:/DSTP/DST_MOD/scripts/dstp/ui_widgets.lua:316-323 — `btn:SetOnClick(function() ... if cmd.callback and UIWidgets._callback_fn then UIWidgets._callback_fn(cmd.callback, cmd.id) end end)`.
[b] Generic-tree node click (text/icon/image/button rendered by RenderNode) goes through `ctx.callback_fn(cb, ctx.root_id)` — ui_widgets.lua:528 (MaybeClickable: `if ctx.callback_fn then ctx.callback_fn(cb, ctx.root_id) end`) and ui_widgets.lua:687 (tree `button`: `if cb and ctx.callback_fn then ctx.callback_fn(cb, ctx.root_id) end`). `ctx.callback_fn` is wired to `UIWidgets._callback_fn` in CreateTree at ui_widgets.lua:760.
[c] The single callback fn `_callback_fn` is SET by modmain via `UIWidgets.SetCallbackHandler` — defined at ui_widgets.lua:965-967 (`function UIWidgets.SetCallbackHandler(fn) UIWidgets._callback_fn = fn end`).
[d] modmain wires that handler at E:/DSTP/DST_MOD/modmain.lua:158-165: `UIWidgets.SetCallbackHandler(function(callback_name, widget_id) if MOD_RPC[modname]['UICallback'] then SendModRPCToServer(MOD_RPC[modname]['UICallback'], callback_name, widget_id, '') end; if RulesEngine then RulesEngine.OnUIButtonClick(callback_name, widget_id, nil) end end)`.     - (2) WHAT IDENTIFIES WHICH WIDGET WAS CLICKED:
Two identifiers travel with every click: `callback` (a.k.a. callback_name) and `widget_id`.
- `callback` is the per-widget callback STRING the flow author set (e.g. 'buy_log', 'open', 'buy:spear'). It is the primary router key.
- `widget_id` is the widget's id. NOTE the asymmetry: for the dedicated `button` widget it passes the actual widget id (`cmd.id`, ui_widgets.lua:321); for generic-tree nodes it passes `ctx.root_id` = the TREE's group/id (ui_widgets.lua:528,687; root_id set in CreateTree at ui_widgets.lua:760 = `cmd.group or cmd.id`), NOT the individual sub-node id. So inside one tree, distinguish buttons by `callback`, not `widget_id`.
- The server RPC handler packs both into the event with redundant keys (modmain.lua:54-61): `callback`, `callback_name` (alias 'for rules/schema consistency'), `widget_id`, plus optional `callback_data`.
- A flow READS them via the trigger context. event.data is spread into triggerData (FlowEngine.ts:421-431) and stored as `context.trigger` (FlowEngine.ts:432-433), so flows use `{{trigger.callback}}` / `{{trigger.callback_name}}` / `{{trigger.widget_id}}` (or via the trigger's alias, e.g. `{{cb.callback}}` when alias='cb' — FlowEngine.ts:440-441). The output schema documents these fields at nodeOutputSchemas.ts:456-464 (userid, name, callback_name, widget_id, callback_data). Real flows route on `{{cb.callback}}` (e.g. shop-full: condition `{{cb.callback}} equals 'open'`, `starts_with 'buy:'` — see scripts/gen-shop-full.cjs:25-27).
    - (3) WHAT A NEW CLICKABLE text/icon/image MUST CALL (reuse, don't reinvent):
The path is already generic — a new clickable primitive must NOT add a new RPC or a new event. It must:
[a] On the BACKEND tree-builder: emit a `callback` prop on the node. FlowEngine already does this generically for ANY node: E:/DSTP/frontend/app/server/live/FlowEngine.ts:734-737 — `// a callback makes ANY node clickable (emits ui_callback) ... if (p.callback) out.callback = String(r(p.callback))`. So the node's params just need a `callback` field.
[b] In the Lua RenderNode (ui_widgets.lua), call the SHARED helper `MaybeClickable(widget, node, ctx)` after creating the widget — exactly as `text` (ui_widgets.lua:627), `icon` (:645) and `image` (:661) already do. MaybeClickable (ui_widgets.lua:520-536) is a no-op unless `node.callback` is set; when set it wires `widget.OnControl` to fire `ctx.callback_fn(cb, ctx.root_id)` (ui_widgets.lua:524-535) with the same 0.5s debounce as buttons. That single `ctx.callback_fn` call is THE reuse point — it lands on `UIWidgets._callback_fn` (set in CreateTree, ui_widgets.lua:760) → modmain's SendModRPCToServer(UICallback) (modmain.lua:158-165) → the identical server→event→backend→trigger path above.
[c] Net: the only thing a new clickable primitive needs is (i) a `callback` param surfaced to the renderer, and (ii) a `MaybeClickable(widget, node, ctx)` call in its RenderNode branch. No new MOD_RPC handler, no new PushEvent type, no new trigger — it rides `ui_callback` and is read with `{{trigger.callback}}`. (The dedicated top-level `button` widget uses the parallel call `UIWidgets._callback_fn(cmd.callback, cmd.id)` at ui_widgets.lua:320-322 — same fn, but only relevant for standalone CreateButton, not tree primitives.)
  - **uncertainties:** 
    - widget_id asymmetry: for the dedicated `button` widget it is the real widget id (cmd.id), but for generic-tree clickables (text/icon/image/tree-button) it is `ctx.root_id` = the tree's group/id, NOT the individual node id. If a future fix needs per-sub-node identity, the click site passes only `cb` + `root_id` to `ctx.callback_fn(cb, ctx.root_id)` (ui_widgets.lua:528,687) — there is currently no path that carries an individual leaf node's id back to the backend; distinguish buttons within a tree by `callback`, or extend the callback fn/CreateTree to pass a node id if leaf-level identity is required.

## plan

- **consolidationPlan:** Do NOT physically delete the five legacy builders in phase 1 — they are live (see deletableLegacy/mustKeepLegacy). The consolidation is a REDIRECTION, not a deletion, done behind the existing action surface so saved flows and example JSON keep working. Stage it:

STAGE 0 (prep, no behavior change): Confirm the two consumer surfaces are truly disjoint. Legacy flat path = action types ui_notification/ui_label/ui_panel/ui_progress_bar/ui_menu/ui_track in actionTypes.ts:46-92 -> FlowEngine.ts:842-924 emits top-level cmd.type in {notification,label,panel,progress_bar,button} -> CREATORS table (ui_widgets.lua:934-942). Tree path = ui_builder/ui_panel exec + shop JSON `"tree":{...}` -> CreateTree -> RenderNode; the shop's `type:"panel"`/`type:"button"` are TREE nodes, never the legacy builders. So consolidation only has to fold the 5 flat builders into the tree renderer; the tree side already owns panel/button/bar/text.

STAGE 1 (fold label/panel/button/progress_bar into the tree renderer via a shim, keep notification as-is): In ui_widgets.lua, replace CreateLabel/CreatePanel/CreateButton/CreateProgressBar bodies with thin adapters that (a) translate the flat cmd into a one-node tree {type=text|panel|button|bar, ...props}, (b) call CreateTree with that tree + the flat cmd's anchor/x/y/group/id, and (c) for label/button/bar, ALSO wrap the result so the single addressable leaf is reachable by the legacy Update* path. The CREATORS entries for label/panel/button/progress_bar then point at these adapters; UPDATERS entries map to a single generic UpdateFlat that routes to SetProps on the wrapped leaf's byId. Net: one rendering layer (RenderNode) produces every primitive; the legacy tables become a compat veneer that just reshapes flat->tree. This removes the duplicated draw code (CreatePanel bg/border/title/body/close, CreateButton ImageButton+debounce, CreateProgressBar fill math) because those now live ONLY in RenderNode's panel/button/bar branches.

STAGE 2 (close the gaps that block folding, so the adapters are faithful): add to the tree renderer the four features the flat builders have and tree lacks — (1) per-leaf screen anchor: let CreateTree already anchors the whole tree, which is sufficient for a single-node flat label/button/bar adapter (wrap the one node at cmd.anchor) — no per-node anchor needed because each flat widget IS its own tree; (2) tree `bar` inline label: add an optional `label` prop to the bar node (a Text child + patch) so progress_bar's label round-trips; (3) tree `text` HAlign/VAlign: add optional halign/valign props so panel body word-wrap region matches legacy; (4) tree `panel` fixed width/height + title/body string slots: add optional width/height/title/body props to the panel node so UpdatePanel-by-name still works via byId. NOTIFICATION stays its own builder (see mustKeepLegacy) — its MoveTo tween + DoTaskInTime auto-dismiss have no tree equivalent; do NOT try to express it as a tree.

STAGE 3 (only after STAGE 1/2 ship and are verified in-game): the four flat builders are now ~10-line adapters; the duplicated draw code is already gone. Leave the adapters as the compat shim (they are tiny and saved flows depend on them). The win is achieved (no dup) without deleting the public action surface.
- **clickFixPlan:** ROOT CAUSE (confirmed against vanilla): `MaybeClickable` (ui_widgets.lua:520-536) wires `widget.OnControl` + `widget:SetClickable(true)` on bare Text/Image. Per vanilla widget.lua, SetClickable only grants focus/hover routing; base `Widget:OnControl` returns false, and the engine only delivers a click as CONTROL_ACCEPT to a widget that actually CONSUMES it the way Button/ImageButton does (button.lua onclick on CONTROL_ACCEPT). A non-focusable HUD Text/Image attached under HUD.controls never reliably receives OnControl, so the override never fires. This is why text/icon/image clicks don't produce ui_callback today while tree `button` (real ImageButton:SetOnClick) does.

EXACT FIX — adopt the vanilla ImageButton-wrap pattern (the one used at widget.lua:759): rewrite `MaybeClickable(widget, node, ctx)` so that when `node.callback` is set it does NOT touch OnControl/SetClickable. Instead it creates a transparent hit-target ImageButton over the primitive:
  local btn = parent:AddChild(ImageButton("images/global.xml","blank.tex","blank.tex","blank.tex","blank.tex"))
  local w,h = <primitive region>   -- text:GetRegionSize() / image:GetSize() / icon size
  btn.image:ScaleToSize(w, h)      -- size the clickable region to the primitive
  btn:SetPosition(0,0)             -- co-located with the primitive in the same parent widget
  btn:SetOnClick(debounced fire)   -- 0.5s debounce, then ctx.callback_fn(cb, ctx.root_id)
The debounce + ctx.callback_fn(cb, ctx.root_id) reuse is identical to the tree `button` branch (ui_widgets.lua:683-688) and lands on UIWidgets._callback_fn -> modmain SendModRPCToServer(UICallback) -> ui_callback event -> FlowEngine trigger. NO new RPC, NO new event, NO new trigger — exactly as the callback-path audit requires.

IMPLEMENTATION NOTES that matter:
- MaybeClickable currently receives only (widget, node, ctx); it must also get the `parent` widget to AddChild the hit-button, OR each caller (text :627, icon :645, image :661) passes its parent. Cleanest: change signature to MaybeClickable(parent, widget, node, ctx) and have the button be a sibling co-located at the primitive's center.
