# net_string clobber (#3) — design validation

> Saved from workflow `Adversarially validate the design decision for DSTP issue #3 (net_string clobber)`. Raw multi-agent research output;
> see the sibling specs for the distilled conclusions.

## ground

**1.** 
  - **summary:** Reported the exact coalescing behavior of Core.ProcessCommands and the 6 _dstp_ui-writing handlers in commands.lua. ProcessCommands coalesces ONLY `ui_command` (per-userid) and passes every other command type straight to ExecuteCommand. Of the 6 handlers, only `ui_command` participates in coalescing; the other 5 (`ui_broadcast`, `install_rules`, `uninstall_rules`, `set_player_state`, `install_rules_all`) bypass coalescing and each does its own raw `:set()` on `_dstp_ui`. Three handlers (install_rules, uninstall_rules, set_player_state, install_rules_all) read/forward a `seq` field; `ui_command`/`ui_broadcast` do not.
  - **facts:** 
    - COALESCING (core.lua ProcessCommands, lines 229-264): ONLY commands with `cmd.type == "ui_command"` that ALSO have `cmd.data.userid` AND `cmd.data.cmd` are coalesced. Condition at core.lua:240: `if cmd.type == "ui_command" and cmd.data and cmd.data.userid and cmd.data.cmd then`.
    - Coalescing mechanism (core.lua:236-263): builds `ui_by_user` (userid -> list of sub-cmds) preserving first-seen order in `ui_order`. If an incoming ui_command's `cmd.action == "batch"` with `cmd.commands`, its sub-commands are flattened into the per-user list (core.lua:244-245); otherwise the single `c` is appended (core.lua:247).
    - Coalescing flush (core.lua:255-263): per userid, if exactly 1 sub it re-issues a plain `ui_command` with `cmd = subs[1]` (core.lua:258-259); if >1 subs it issues ONE `ui_command` with `cmd = { action = "batch", commands = subs }` (core.lua:260-261). All go through Core.ExecuteCommand.
    - STRAIGHT-PASS (core.lua:249-252): EVERY command that is NOT a qualifying ui_command — including ui_broadcast, install_rules, uninstall_rules, set_player_state, install_rules_all, and any non-UI command — falls to the `else` branch and is sent directly to `Core.ExecuteCommand(cmd)` with no batching. Note: a ui_command MISSING userid or cmd would also fall here, but in practice all 6 special handlers except ui_command bypass coalescing.
    - HANDLER 1 `ui_command` (commands.lua:845-866): PER-PLAYER (FindPlayer(data.userid), commands.lua:850). It does NOT build a fixed cmd table — it encodes `data.cmd` as-is via SafeEncode (commands.lua:859) and `player.player_classified._dstp_ui:set(json_str)` (commands.lua:864). The cmd shape/action is whatever the backend/coalescer supplies (e.g. `{action="batch", commands={...}}` or a single sub-cmd). NO seq field touched by the handler.
    - HANDLER 2 `ui_broadcast` (commands.lua:870-886): BROADCAST-TO-ALL (loops `_G.AllPlayers`, commands.lua:880). Encodes `data.cmd` as-is (commands.lua:875) and sets the same json_str on every player's `_dstp_ui` (commands.lua:882). No fixed action built by handler; uses caller's `data.cmd`. NO seq field.
    - HANDLER 3 `install_rules` (commands.lua:889-899): PER-PLAYER (FindPlayer(data.userid), commands.lua:891). Builds `local cmd = { action = "rules_install", rules = data.rules, seq = data.seq }` (commands.lua:893), SafeEncodes it, and `_dstp_ui:set(json_str)` on that one player (commands.lua:896). READS/SETS `seq` (= data.seq).
    - HANDLER 4 `uninstall_rules` (commands.lua:902-911): PER-PLAYER (FindPlayer(data.userid), commands.lua:904). Builds `local cmd = { action = "rules_uninstall", ids = data.ids, seq = data.seq }` (commands.lua:906), SafeEncodes, and `_dstp_ui:set(json_str)` on that player (commands.lua:909). READS/SETS `seq` (= data.seq).
    - HANDLER 5 `set_player_state` (commands.lua:914-923): PER-PLAYER (FindPlayer(data.userid), commands.lua:916). Builds `local cmd = { action = "state_set", key = data.key, value = data.value, seq = data.seq }` (commands.lua:918), SafeEncodes, and `_dstp_ui:set(json_str)` on that player (commands.lua:921). READS/SETS `seq` (= data.seq).
    - HANDLER 6 `install_rules_all` (commands.lua:926-937): BROADCAST-TO-ALL (loops `_G.AllPlayers`, commands.lua:931). Builds `local cmd = { action = "rules_install", rules = data.rules, seq = data.seq }` (commands.lua:928), SafeEncodes ONCE, then sets the same json_str on every player's `_dstp_ui` (commands.lua:933). READS/SETS `seq` (= data.seq). Note: same `action="rules_install"` as HANDLER 3 but broadcast.
    - SEQ SUMMARY: 4 of the 6 handlers read+forward a `seq` field (install_rules:893, uninstall_rules:906, set_player_state:918, install_rules_all:928), all as `seq = data.seq`. The other 2 (ui_command, ui_broadcast) never reference `seq` — they pass `data.cmd` through verbatim, so seq is only present if the backend put it inside data.cmd.
    - CRITICAL CLOBBER NOTE: All 5 non-coalesced handlers (ui_broadcast, install_rules, uninstall_rules, set_player_state, install_rules_all) call `_dstp_ui:set()` directly. Since `_dstp_ui` is a single-value net_string (core.lua:232-235 comment), if more than one of these (or one of these PLUS a coalesced ui_command) targets the SAME player in the SAME sync, only the LAST `:set()` survives. ProcessCommands only coalesces `ui_command`; it does NOT merge these 5 handlers' writes with each other or with the ui_command batch.
  - **uncertainties:** 
    - The task referenced these handlers as living around commands.lua line 845-938 — confirmed exact: all 6 are within commands.lua:845-937. The task also referenced core.lua ProcessCommands ~line 229 — confirmed exact at core.lua:229-264.
    - `SafeEncode`, `FindPlayer`, `Log`, `LogError`, `config` are upvalues/locals in commands.lua not shown in the read window (defined earlier in the file or injected via Init); their bodies were not inspected but their call semantics are clear from usage.
    - Whether the BACKEND ever emits ui_broadcast / install_rules_all / set_player_state for the same player alongside ui_command in one sync (which would trigger the single-value net_string clobber) is a backend-side question not answerable from these two Lua files.
**2.** 
  - **summary:** The `dstp_ui_dirty` net_string listener in modmain.lua (lines 141-177) decodes the JSON payload from `inst._dstp_ui` and routes the WHOLE decoded command to exactly one side based on a prefix check of the TOP-LEVEL `cmd.action` string. A `batch` command has action == "batch", which does NOT start with "rules_" or "state_", so the entire batch is handed to UIWidgets.ProcessCommand — UIWidgets then loops over cmd.commands and processes each sub-command itself. The modmain router never inspects sub-command actions; a "batch" containing rules_/state_ sub-commands would be misrouted (all sub-commands go through UIWidgets, which has no rules_/state_ handlers). There is NO seq/dedup logic anywhere in modmain.lua.
  - **facts:** 
    - Decode + guard: `local cmd_str = inst._dstp_ui:value()` then empty-check; JSON decode via `local ok, cmd = GLOBAL.pcall(GLOBAL.json.decode, cmd_str)` and bail if `not (ok and cmd and cmd.action)` (modmain.lua:142-143, 162-163).
    - (1) RulesEngine-vs-UIWidgets decision is a 6-char prefix check on the TOP-LEVEL action: `local act = tostring(cmd.action)` then `if act:sub(1, 6) == "rules_" or act:sub(1, 6) == "state_"` routes to RulesEngine (lazy-required/Init'd), `else` routes to UIWidgets (modmain.lua:165-176). Comment at line 165: 'Route rules_* / state_* to RulesEngine, others to UIWidgets'.
    - (2) For `cmd.action == "batch"`: action 'batch' does not match the `rules_`/`state_` prefixes, so the entire decoded cmd goes to the ELSE branch -> `UIWidgets.ProcessCommand(cmd)` (modmain.lua:174-175). The modmain router does NOT inspect any sub-command actions; it only ever reads the single top-level `cmd.action`.
    - (2 cont.) The batch fan-out happens inside UIWidgets, not modmain: `elseif cmd.action == "batch" then ... for _, sub_cmd in ipairs(cmd.commands) do UIWidgets.ProcessCommand(sub_cmd) end` (ui_widgets.lua:1089-1095). So every sub-command is processed by UIWidgets recursively; sub-commands are never re-evaluated against the modmain rules_/state_ prefix router, and UIWidgets has no rules_/state_ handlers (unknown actions hit the `else` log at ui_widgets.lua:1096-1097).
    - (3) No seq-dedup behavior exists: a case-insensitive search for 'seq' across modmain.lua returns zero matches. The listener processes whatever the net_string currently holds on each `dstp_ui_dirty` event with no sequence number, no last-seen tracking, and no duplicate suppression (modmain.lua:141-177).
  - **uncertainties:** 
    - The task framed this as the listener routing 'to RulesEngine vs UIWidgets'; I confirmed the batch fan-out lives in ui_widgets.lua (line 1089) since modmain only routes the top-level action — this is the mechanism, not a proposed fix.
**3.** 
  - **summary:** Both ProcessCommand functions implement seq-dedup with a PER-SIDE module-local _seq counter (one in ui_widgets.lua, an independent one in rules_engine.lua). UIWidgets.ProcessCommand handles a "batch" action but recurses into itself for each sub-command — it does NOT re-route sub-commands through modmain's prefix router, so a rules_install sub-command inside a UI batch falls through to the UIWidgets "unknown action" else-branch and is silently dropped (never reaches RulesEngine). RulesEngine.ProcessCommand has NO "batch" action. Because each side advances its own _seq to the envelope's seq, coalescing many commands under ONE shared seq value and sending them on both channels would cause each side to process the first arrival and then drop every later command carrying the same-or-lower seq.
  - **facts:** 
    - (1a) UIWidgets.ProcessCommand DOES handle 'batch': E:/DSTP/DST_MOD/scripts/dstp/ui_widgets.lua:1089 `elseif cmd.action == "batch" then`, body lines 1090-1095.
    - (1b) Inside the batch branch it calls UIWidgets on each sub-command, NOT a re-route: ui_widgets.lua:1092 `for _, sub_cmd in ipairs(cmd.commands) do` and 1093 `UIWidgets.ProcessCommand(sub_cmd)`. It recurses on the SAME UIWidgets dispatcher.
    - (1c) UIWidgets.ProcessCommand has no case for rules_install/rules_*/state_*; any such sub-command falls to the else/unknown-action branch: ui_widgets.lua:1096 `else` and 1097 `Log("unknown action '" .. tostring(cmd.action) .. "'")`. So a rules_install inside a UI batch would hit 'unknown action' and be dropped — it is NOT handled.
    - (1d) Routing to RulesEngine vs UIWidgets happens ONLY at the top level in modmain by action-prefix string match, before ProcessCommand is called: E:/DSTP/DST_MOD/modmain.lua:166-176. modmain.lua:167 `if act:sub(1, 6) == "rules_" or act:sub(1, 6) == "state_" then` -> RulesEngine.ProcessCommand (line 173), else UIWidgets.ProcessCommand (line 175). A top-level action of 'batch' does not match the rules_/state_ prefix, so the whole envelope goes to UIWidgets (line 175); the batch branch never consults this router again, so nested sub-commands are never re-routed.
    - (2) RulesEngine.ProcessCommand does NOT handle 'batch'. Its action dispatch (rules_engine.lua:441-463) covers only: rules_install (441), rules_uninstall (447), rules_clear (453), state_set (455), state_delete (457), state_clear (459), else -> unknown action (461-462). No 'batch' case.
    - (3a) Seq-dedup is PER-SIDE (independent module-local counters), not shared. UIWidgets: declared at ui_widgets.lua:8 `local _seq = -1  -- last processed sequence number (dedup net_string replays)`; used at 1068-1070 (`if cmd.seq then` / `if cmd.seq <= _seq then return end` / `_seq = cmd.seq`).
    - (3b) RulesEngine has its OWN separate counter: declared at rules_engine.lua:15 `local _seq = -1`; used at 434-436 (`if cmd.seq then` / `if cmd.seq <= _seq then return end` / `_seq = cmd.seq`). The two _seq variables are distinct module locals.
    - (3c) Both use the same '<=' gate (skip if cmd.seq <= last) and advance to cmd.seq. Therefore, coalescing many commands under ONE identical seq value on a single side: the first one with that seq sets _seq, and every subsequent command carrying that same seq (or any lower seq) satisfies `cmd.seq <= _seq` and is dropped (return) — all but one would be dropped. The dedup gate is at the top of ProcessCommand, so it applies to the batch ENVELOPE; sub-commands inside a batch carry no seq of their own in the iterated loop, so within a single batch the inner UIWidgets.ProcessCommand calls are not individually deduped (sub_cmd.seq is nil unless the sender set it).
  - **uncertainties:** 
    - Whether the backend ever sets cmd.seq on individual sub-commands inside a batch was not inspected (backend code not read); the Lua dedup only acts on whatever cmd.seq is present on each command object passed to ProcessCommand. If sub-commands carry their own seq, the recursive UIWidgets.ProcessCommand would also seq-gate them against the same shared UIWidgets _seq.
    - I did not inspect the backend TypeScript that builds these net_string payloads, so I cannot confirm whether rules_* commands are ever actually packed inside a 'batch' envelope in practice (the question's hypothetical) — I only confirmed the Lua-side behavior if they were.

## verdicts

**1.** 
  - **approach:** Approach A — server coalesces ALL _dstp_ui-writing commands (ui_command, ui_broadcast, install_rules, uninstall_rules, set_player_state, install_rules_all) per-player into ONE {action:"batch", commands:[...]} :set once; AND modmain's dstp_ui_dirty router is made batch-aware so each sub-command is routed by its OWN action prefix (rules_/state_ -> RulesEngine, else -> UIWidgets).
  - **fatalFlaws:** 
    - SEQ-DEDUP COLLAPSE (the killer). After re-routing, each side's ProcessCommand applies its <= seq gate at the ENVELOPE top, but sub-commands carry their OWN independent seqs. Today seq = Date.now() generated PER COMMAND on the backend (FlowEngine.ts: install_rules:956, uninstall_rules:963, set_player_state:970, ui_menu batch:900). When the new batch-aware modmain hands a rules_* sub-command to RulesEngine.ProcessCommand, that sub-command's seq is whatever data.seq the handler stamped (commands.lua:893/906/918) — but the coalesced UI sub-commands have NO seq at all (single create/set/destroy/clear never set seq; only ui_menu does). So the dedup gate is applied INCONSISTENTLY: UI subs are never deduped, rules subs are deduped against RulesEngine's single shared _seq using Date.now() values that COLLIDE within one engine tick.
    - MILLISECOND seq COLLISION. Date.now() is 1ms resolution. Two rule_install / rule_set_state actions fired in the SAME engine tick (same flow run, synchronous JS) produce IDENTICAL seq. On the client, RulesEngine processes the first (sets _seq=S), then every later sub-command with the same S satisfies cmd.seq <= _seq and is DROPPED via `return`. Coalescing many rules commands into one batch makes this far more likely than today (today they arrive as separate envelopes across syncs, but even today two same-ms commands collide). Approach A turns an edge case into the common case for multi-rule batches.
    - ENVELOPE-vs-SUB seq AMBIGUITY. core.lua coalescer would have to put SOME seq on the outer {action:'batch'} envelope. If it does, UIWidgets.ProcessCommand (which is still the entry for the whole batch envelope when top action=='batch') advances ITS _seq to the envelope seq at ui_widgets.lua:1068-1070 BEFORE the new router even sees sub-commands — meaning a future legitimate single ui_command with a LOWER Date.now()-derived seq (clock skew / batch seq from a later ms) interacts unpredictably with the UI _seq. The two _seq counters (UIWidgets ui_widgets.lua:8, RulesEngine rules_engine.lua:15) now get cross-contaminated by the same physical batch.
    - ROUTING REGRESSION RISK: today UIWidgets.ProcessCommand already fans out 'batch' internally (ui_widgets.lua:1089-1095). If modmain is made batch-aware to re-route per sub-command, you MUST also delete/neutralize the internal UIWidgets batch fan-out, otherwise a pure-UI batch is processed TWICE (once by the new modmain loop, once by UIWidgets' own batch branch) — double-create widgets, duplicate notifications. The approach as stated only mentions changing modmain, not removing the UIWidgets batch branch.
  - **breakingCases:** 
    - SAME-PLAYER MIXED, SAME TICK: A flow does rule_install(user=joe) then ui_notification(user=joe) then rule_set_state(user=joe) in one run. core.lua coalesces into ONE batch for joe. Backend stamped install_rules seq=1733000000000 and set_player_state seq=1733000000000 (same ms). New modmain routes sub[0] rules_install -> RulesEngine: _seq=1733000000000. Routes sub[2] state_set: 1733000000000 <= 1733000000000 -> DROPPED. Player state never set. Silent data loss.
    - BROADCAST + PER-PLAYER COLLISION: same sync emits install_rules_all (broadcast, seq=T) AND a per-player ui_command(create) for joe. core.lua per-player coalescing must merge joe's broadcast copy with his per-player UI into one batch. But ui_broadcast/install_rules_all are NOT keyed by userid in ProcessCommands today (they fall to the else straight-pass at core.lua:249-252) — Approach A requires REWRITING the coalescer to expand broadcasts into per-player entries against _G.AllPlayers, which the BACKEND core.lua cannot do (it doesn't know the live player list; that's the mod's job via _G.AllPlayers). The coalescer literally cannot fold a broadcast into per-user batches server-side. This forces the broadcast handlers to STILL bypass coalescing -> the clobber survives for broadcast+per-player same-player overlap.
    - TWO ui_menu IN ONE TICK to same player: each ui_menu sets its own batch seq=Date.now() (FlowEngine.ts:900), same ms. After Approach A flattens both menus' sub-commands into the outer coalesced batch (core.lua:244-245 already flattens nested batches), the OUTER batch needs one seq; whichever inner seq wins, the UI _seq is advanced once and a re-delivered net_string replay (the very thing seq defends against) of EITHER menu is now mis-deduped.
    - RAPID RE-DELIVERY: net_string replays the last value on reconnect/dirty re-fire. With a coalesced batch carrying a Date.now() envelope seq, if the same batch is re-fired the dedup works; but if a DIFFERENT later command for the same player produced a seq LOWER than a clock-skewed earlier batch (Date.now() is not monotonic across NTP adjustments), the later real command is silently dropped by the <= gate.
  - **clientChangeNeeded:** FORCED Lua changes (non-trivial): (1) modmain.lua:165-176 router must iterate cmd.commands when cmd.action=='batch' and route EACH sub by its own prefix to RulesEngine vs UIWidgets — but this ALSO requires removing/neutralizing the existing internal batch fan-out in ui_widgets.lua:1089-1095 to avoid double-processing UI subs. (2) The seq-dedup model must be redesigned: the single envelope <= seq gate (ui_widgets.lua:1068-1070, rules_engine.lua:434-437) is incompatible with a mixed batch whose subs have heterogeneous/absent/colliding seqs. Either move dedup to PER-SUB-COMMAND with a per-(side) monotonic counter, or have the backend stamp a single strictly-increasing batch seq and dedup ONLY the envelope (and stop relying on per-command Date.now()). (3) core.lua ProcessCommands must be rewritten to key install_rules/uninstall_rules/set_player_state by userid into ui_by_user — straightforward — BUT broadcasts (ui_broadcast/install_rules_all) CANNOT be folded server-side because the live player list lives in the mod (_G.AllPlayers), so the broadcast path either stays un-coalesced (clobber survives) or the mod must expand broadcasts into per-player batches client-side before :set (a deeper mod change).
  - **reloadRisk:** HIGH. Touches three hot Lua files (core.lua ProcessCommands, modmain.lua listener, ui_widgets.lua + rules_engine.lua dedup). modmain.lua changes require a full DST client restart (modmain runs at mod load, not hot-reloadable). The dedup redesign means the module-local _seq counters (ui_widgets.lua:8, rules_engine.lua:15) reset to -1 on reload while the backend's Date.now() seq keeps climbing — after a mod reload mid-session the first post-reload batch is accepted (any seq > -1), which is fine, but if the backend is ALSO restarted its Date.now() could go backward relative to a client that wasn't reloaded → stale-seq drops. Cross-restart seq coherence is fragile precisely because seq is wall-clock, not a per-session counter.
  - **verdict:** broken
  - **score:** 3
**2.** 
  - **approach:** Approach B: split rules/state onto their own net_string (_dstp_rules) separate from _dstp_ui. UI coalesces into _dstp_ui, rules/state coalesce into _dstp_rules. By construction the two families cannot clobber. Requires a new net_string on player_classified + a new client listener.
  - **fatalFlaws:** 
    - INTRA-FAMILY CLOBBER IS NOT SOLVED — Approach B only adds a SECOND channel; it does not add coalescing to the 4 rules/state handlers. Per the ground truth, core.lua ProcessCommands coalesces ONLY cmd.type=='ui_command' (core.lua:240). install_rules / uninstall_rules / set_player_state / install_rules_all all fall to the else branch (core.lua:249-252) and each does its OWN raw :set(). If you simply re-point those 4 handlers' :set() from inst._dstp_ui to inst._dstp_rules, two rules/state commands for the SAME player in ONE sync still both :set() _dstp_rules in the same frame and the LAST still wins. You have moved the clobber, not removed it. To actually fix it you ALSO have to extend ProcessCommands to coalesce the rules/state family into a batch on _dstp_rules — which is strictly MORE work than Approach 'coalesce everything into one channel', not less. The 'never clobber by construction' claim is false: it only separates UI-vs-rules, not rules-vs-rules.
    - NEW net_string CHANGES THE player_classified NETVAR LAYOUT — _dstp_pm, _dstp_ui, _dstp_keys are declared positionally in one AddPrefabPostInit('player_classified') (modmain.lua:102/105/110). DST netvars are positional: the spec dynamic-data-bindings.md explicitly warns netvars must be declared in a stable order on BOTH sides or they desync/crash. Adding a 4th net_string is a wire-format change to a core replica prefab. Every connected client MUST run the new mod build simultaneously; a server on the new build + any client on the old build (or vice-versa during a Workshop rollout) gets a netvar-count mismatch on player_classified → silent misread or client crash. This is a hard mod-reload / version-lock hazard that the single-channel fix (which touches only payload shape, not netvar count) does NOT have.
    - RULES_ENGINE BATCH GAP IS UNTOUCHED — RulesEngine.ProcessCommand has NO 'batch' action (rules_engine.lua:441-463: only rules_install/uninstall/clear, state_set/delete/clear, else=unknown). So the moment you DO add coalescing to _dstp_rules (which you must, see flaw 1), you'll emit {action:'batch', commands:[...]} on _dstp_rules and the new rules listener will hit the else→'unknown action' branch and DROP the entire batch. Approach B forces you to ALSO add a 'batch' case to rules_engine.lua. The approach as described ('rules/state coalesce into the rules net_string') is non-functional without a rules_engine.lua code change it doesn't mention.
  - **breakingCases:** 
    - SEQ COLLISION ACROSS CHANNELS (most subtle): backend stamps seq:Date.now() on rules/state (FlowEngine.ts:956,963,970) AND on the UI batch envelope (FlowEngine.ts:900) from the SAME wall clock. With separate channels you have separate _seq counters (ui_widgets.lua:8 and rules_engine.lua:15, both independent, both '<=' gated). That part is fine. BUT: if two rules/state commands for one player are emitted in the same millisecond (e.g. rule_install + rule_set_state in one flow, both Date.now()), they get the SAME seq. After you add coalescing they may arrive as separate :set() (until you also batch them) OR as a batch. If separate: the second :set() clobbers the first on _dstp_rules (flaw 1). If you DON'T batch and instead rely on replay: rules_engine _seq advances to seq on the first, then the second with seq<=_seq is DROPPED (rules_engine.lua:435). Concrete break: flow does rule_install (seq=1717000000000) then rule_set_state (seq=1717000000000) in the same tick → only ONE survives regardless of channel split. Approach B does nothing for this; it's a same-ms seq collision orthogonal to the channel.
    - MIXED PER-PLAYER + BROADCAST IN ONE SYNC: install_rules_all (broadcast, commands.lua:926, loops AllPlayers) AND a per-player install_rules for player X arrive in one /dst/sync. Both are rules-family → both now target _dstp_rules. Neither is coalesced (only ui_command is, core.lua:240). For player X: install_rules_all :set(json_A) then install_rules :set(json_B) both fire on _dstp_rules same frame → only json_B survives → player X never gets the broadcast rules. Approach B's separate channel does NOT help because BOTH writers are in the rules family and ProcessCommands still doesn't coalesce them. This is the exact clobber from issue #3, just relocated to _dstp_rules.
    - UI + RULES MIXED (the case Approach B IS meant to fix) — this one it DOES fix: ui_command (→_dstp_ui) + install_rules (→_dstp_rules) for the same player in one sync now write to different net_strings, both survive. Confirmed this specific case works. But it's the ONLY case it fixes, and it's the narrowest of the four breaking scenarios.
    - ORDERING IS LOST BETWEEN CHANNELS: a flow that does ui_clear then rule_install expecting the UI gone before rules attach now races — two net_strings fire two independent dstp_ui_dirty / dstp_rules_dirty events with no cross-channel ordering guarantee. With one coalesced batch on a single channel, sub-command order is preserved (ui_widgets.lua:1092 ipairs). Approach B trades a clobber bug for an ordering non-determinism between the two channels.
  - **clientChangeNeeded:** FORCES MULTIPLE Lua changes, more than the single-channel fix: (1) modmain.lua — add a 4th net_string `inst._dstp_rules = net_string(inst.GUID, 'dstp.rules', 'dstp_rules_dirty')` in the AddPrefabPostInit('player_classified') block (changes player_classified netvar layout — version-lock hazard) and register a NEW ListenForEvent('dstp_rules_dirty') that routes straight to RulesEngine. (2) core.lua ProcessCommands — to actually stop intra-rules clobber you must add a SECOND coalescer pass building rules_by_user and flushing one batch per player onto the rules channel (currently nothing coalesces the rules family). (3) rules_engine.lua — add a 'batch' action case (currently absent, rules_engine.lua:461 drops it as unknown) or the coalesced batch is silently dropped. (4) backend FlowEngine.ts — the 4 rules/state pushCommand calls + the relay/dst.routes :set() target selection must learn which net_string each command type goes to. Net: 4 files touched vs the single-channel fix which touches only core.lua ProcessCommands (generalize coalescing to all _dstp_ui commands) + reuses the existing batch handler that BOTH sides already have for ui_widgets.
  - **reloadRisk:** HIGH. Adding a net_string to player_classified changes the replica's netvar count/order — a wire-format change to a core engine prefab. DST netvars are positional and must match server↔client (dynamic-data-bindings.md desync warning). During a Workshop rollout, any server/client version skew yields a player_classified netvar mismatch → silent corruption or client crash on join. Server and ALL clients must update to the same mod build atomically. The single-channel fix changes only the JSON payload shape inside the EXISTING _dstp_ui net_string — no netvar layout change, so an old client just sees a richer batch (and already has the batch handler at ui_widgets.lua:1089), degrading gracefully. Approach B's reload risk is categorically worse.
  - **verdict:** broken
  - **score:** 3
**3.** 
  - **approach:** Approach C: server-only — coalesce into SEPARATE per-family batches (one UI batch, one rules batch) emitted as two sequential :set() calls on the same _dstp_ui in the same frame, no client change.
  - **fatalFlaws:** 
    - DOES NOT SOLVE THE ISSUE — it is the bug. The whole defect (core.lua:232-235 comment, confirmed) is that _dstp_ui is a single-value net_string: multiple :set() in one frame => only the LAST survives. Approach C reduces N writes to 2 writes, but 2 > 1, so the first batch (e.g. the UI batch) is unconditionally clobbered by the second (the rules batch). It converts a 6-way clobber into a guaranteed 2-way clobber. No improvement on the core symptom.
    - Two :set() on the SAME net_string in the SAME frame is NOT two deliveries. modmain.lua:142 reads inst._dstp_ui:value() — the CURRENT held value. After :set(A); :set(B) on the master in one frame, the replicated value the client ever sees is B. The first family's batch (A) is dropped before it can be decoded. The dstp_ui_dirty listener has no queue, no accumulation, no per-frame buffer (only one net_string declared at modmain.lua:105, one listener at 141).
    - It still requires the backend to know which family each command belongs to and emit them in a deterministic order; whichever family the backend writes FIRST is the one that gets silently lost. So under Approach C, e.g. UI updates would always be eaten whenever rules/state are co-delivered to the same player in the same sync — a silent, data-dependent loss with no error.
    - The seq-dedup makes it strictly worse, not a safety net. UIWidgets._seq (ui_widgets.lua:8/1068-1070) and RulesEngine._seq (rules_engine.lua:15/434-436) are INDEPENDENT module-locals, so two separate batches with their own seqs don't cross-block — but that doesn't rescue C, because the loss happens at the net_string layer (value B overwrites A) BEFORE either ProcessCommand ever runs. The client never receives A to dedup in the first place.
  - **breakingCases:** 
    - Single /dst/sync queues ui_command(create panel) + install_rules for SAME player. core.lua coalesces ui_command into a UI batch (handler at commands.lua:845 -> :set(UI_json)); install_rules handler (commands.lua:889) -> :set(RULES_json) on the same player.player_classified._dstp_ui in the same frame. Whichever runs second wins; the other is lost. C does nothing because both handlers still each call raw :set() on the one net_string.
    - Single sync: ui_command + set_player_state to SAME player. Same clobber: commands.lua:864 :set(UI) vs commands.lua:921 :set(state). state_set carries seq (data.seq) but seq is irrelevant — the UI write or the state write is destroyed at the net_string before the client decodes anything.
    - Broadcast collision: ui_broadcast (commands.lua:870-886, loops AllPlayers) PLUS install_rules_all (commands.lua:926-937, loops AllPlayers) in one sync. Every player gets :set(UI) then :set(rules) (or vice-versa). For EVERY player one of the two is clobbered. Approach C's 'two batches' is exactly this scenario and it fails for the entire server at once.
    - Mixed per-player + broadcast: player P gets a targeted ui_command (UI batch) AND is also in AllPlayers for an install_rules_all in the same sync. P receives two :set() in the frame; one is lost. C cannot merge a per-player UI batch with a broadcast rules batch because they are produced by different handlers writing the same net_string.
    - Three+ families: ui_command + install_rules + set_player_state to one player. C only defines TWO batches (UI, rules-family). If 'rules-family' lumps install_rules+set_player_state into one rules batch, that requires a NEW unified rules envelope action the client doesn't have (RulesEngine.ProcessCommand has no 'batch' case — rules_engine.lua:441-463), so it forces a client change, contradicting C's 'no client change'. If instead each handler still writes separately, you have 3 :set() and 2 survivors lost.
  - **clientChangeNeeded:** As stated ('no client change'), Approach C is simply broken — two :set() per frame cannot both be observed by modmain.lua:142's single :value() read. To make ANY variant of 'separate batches' work you are forced into client changes that C claims to avoid: EITHER (a) wrap everything in ONE mixed envelope and make the client fan sub-commands to the correct side — but UIWidgets.ProcessCommand's 'batch' branch (ui_widgets.lua:1089-1095) recurses into UIWidgets only and has no rules_/state_ handlers, so rules sub-commands hit the 'unknown action' else (ui_widgets.lua:1096-1097) and are silently dropped; you'd need to add sub-command re-routing in the client (modmain prefix router or a UIWidgets dispatch that re-consults rules_/state_). OR (b) add a real per-frame ACCUMULATION buffer on the client so two :set()/two dirty events are both captured — but net_string only retains the last value, so even a buffering listener can miss the first value entirely; this fundamentally cannot be done client-side without a second net_string channel or a single-envelope protocol. Net: C as scoped (server-only, no client change) is impossible to make correct; the minimal correct fix is the single all-command batch already recommended in mod-audit-2026-06.md:42-43 PLUS a client batch fan-out that re-routes rules_/state_ sub-commands to RulesEngine.
  - **reloadRisk:** If shipped server-only (the literal C): ZERO mod-reload needed (backend-only change) — but the feature is still broken, so 'low reload risk' is meaningless. If forced into the required client fix (sub-command re-routing in modmain/ui_widgets so a single mixed batch fans rules_/state_ to RulesEngine), that IS a Lua change to ui_widgets.lua and/or modmain.lua, forcing every DST host to redownload/reinstall the mod and reload (copy to mods folder, Workshop bump). So the honest reload risk is HIGH: C cannot ship correctly without a coordinated client Lua change, which is exactly what C promised to avoid — and a mismatched backend (new mixed envelope) against an old client silently drops rules sub-commands (unknown action), a worse, harder-to-diagnose failure than the current clobber.
  - **verdict:** broken
  - **score:** 1

## decision

- **chosen:** Hardened single-channel coalesce-all (Approach A's channel choice, but with all three of A's fatal flaws fixed). Coalesce EVERY _dstp_ui-writing command per-player into ONE {action:"batch", commands:[...]} envelope set once per player per sync, but: (1) replace per-command Date.now() seq with a single backend-assigned strictly-monotonic per-(serverId) counter stamped ONLY on the outer batch envelope, dedup ONLY the envelope on the client against ONE shared _seq; (2) expand broadcasts (ui_broadcast/install_rules_all) to per-player entries INSIDE the mod's Core.ProcessCommands where _G.AllPlayers is live, folding them into each player's batch; (3) make the client batch-aware by re-routing each sub-command by its OWN action prefix (rules_/state_ -> RulesEngine, else -> UIWidgets) from a single dispatcher, add a 'batch' case to rules_engine.lua, and remove the now-redundant internal UIWidgets batch fan-out so a sub-command is processed exactly once.
- **why:** On the EVIDENCE, the channel decision is forced. Approach B adds a 4th net_string to player_classified; the mod declares _dstp_pm/_dstp_ui/_dstp_keys positionally in one AddPrefabPostInit (modmain.lua:100-110) and dynamic-data-bindings.md warns netvars are positional and must match server<->client or they desync/crash. The relay model means many independently-updating DST hosts, so a Workshop rollout guarantees server/client version skew, giving B a hard wire-format version-lock crash the single-channel fix structurally cannot have: the single channel changes only the JSON payload inside the EXISTING _dstp_ui, and an un-updated client already has a batch handler (ui_widgets.lua:1089) so it degrades gracefully. Approach C is physically impossible as scoped: modmain.lua:142 reads inst._dstp_ui:value() (the single held value); two :set() in one frame means the client only ever observes the second, so C converts a 6-way clobber into a guaranteed 2-way clobber — it IS the bug. That leaves the single-channel family (A). A was scored broken only for three SOLVABLE defects, all rooted in one bad choice — using wall-clock Date.now() as the dedup key. The code confirms the queue is FIFO (DSTStateStore commandQueues .push, line 181) and ProcessCommands iterates in order, so flow-emit ORDER is already the real sequencer; wall-clock seq is unnecessary and is exactly what collides at 1ms within a tick (FlowEngine.ts:900/956/963/970 all Date.now()). Replacing it with a monotonic counter on the envelope and deduping only the envelope eliminates A's SEQ-DEDUP-COLLAPSE and MILLISECOND-COLLISION flaws outright. A's broadcast-fold objection (core.lua cant expand broadcasts, doesnt know AllPlayers) is wrong about WHERE the work happens: the coalescer lives in the MOD's Core.ProcessCommands (core.lua:229), which DOES have _G.AllPlayers (used at commands.lua:880/931), so broadcast expansion is trivially available there. A's routing-regression objection (double fan-out) is real but is a one-line deletion of the UIWidgets internal batch branch once modmain owns fan-out. So the single-channel approach is the only one whose flaws are all repairable without a wire-format/version-lock hazard, and it touches the fewest hot files with the safest backward-compat story.
- **risks:** 
  - modmain.lua is NOT hot-reloadable (runs at mod load); the routing/fan-out change forces a full DST client restart and a Workshop bump — every host must redownload. Mitigation: keep the envelope shape backward-tolerant so a stale client (old modmain that hands the whole batch to UIWidgets, whose existing batch branch at ui_widgets.lua:1089 fans sub-commands into UIWidgets) still processes UI sub-commands; only rules_/state_ subs are lost on a stale client, same as today's clobber — never worse.
  - Monotonic seq must be per-(serverId/shard) and survive HMR on the backend (like WorkflowInstanceStore lives on globalThis). If it resets to a value <= one a long-lived client already saw, the client's <= gate silently drops the next batch. Mitigation: seed the counter from Date.now() at boot then ++ (monotonic AND wall-clock-ordered across restarts), OR have the client reset its _seq on player_classified re-attach. Cross-restart coherence is the residual sharp edge.
  - After a mid-session mod reload the client module-local _seq resets to -1 while the backend counter keeps climbing — first post-reload batch is accepted (any seq > -1), which is correct; but a backend restart that lowers the counter below an un-reloaded client's _seq drops batches. Same mitigation (seed from wall clock).
  - Per-player batch can grow large if many broadcasts + per-player commands target one player in one sync; net_string is a single string — a very large JSON could exceed practical net_string size. Mitigation: bounded by commands-per-sync (already small); document a soft cap and log if a single player's batch exceeds N sub-commands.
  - Removing the UIWidgets internal batch fan-out is load-bearing: if modmain fan-out ships but the UIWidgets branch is NOT removed, pure-UI batches double-process (duplicate widgets/notifications). These two edits MUST land together.
  - ui_command/ui_broadcast subs carry NO seq today (only the envelope will). Since dedup moves to the envelope ONLY, individual sub-commands must NOT be seq-gated (sub_cmd.seq must be nil/ignored) or a stray sub seq would corrupt the shared _seq. Client dedup must read seq from the envelope exclusively.
- **mustHandle:** 
  - SEQ DEDUP UNDER COALESCE: assign ONE strictly-monotonic seq to the OUTER batch envelope only (backend per-server counter, seeded from Date.now() at boot so it is monotonic across restarts). The client dedups ONLY the envelope against ONE shared _seq; sub-commands carry no seq and are never individually gated. This kills the same-millisecond Date.now() collision that drops co-tick rules+state.
  - BROADCAST EXPANSION: in the MOD's Core.ProcessCommands, treat ui_broadcast and install_rules_all as broadcast: iterate _G.AllPlayers and append the broadcast sub-command into EACH player's per-user batch list (the same ui_by_user map, now keyed for ALL families, not just ui_command). Do NOT leave any family on the straight-pass else branch — that is the relocated-clobber trap that sank A and B.
  - FULL-FAMILY COALESCE: extend the coalescer key set from {ui_command} to ALL six _dstp_ui-writing types (ui_command, ui_broadcast, install_rules, uninstall_rules, set_player_state, install_rules_all). Per-player handlers fold by userid; broadcast handlers fold into every player. Each non-ui family's sub-command must be the SAME shape the client router expects: install_rules -> {action:'rules_install', rules}, uninstall_rules -> {action:'rules_uninstall', ids}, set_player_state -> {action:'state_set', key, value}. These shapes already exist in commands.lua:893/906/918 — build them in the coalescer instead of in the per-handler :set().
  - ORDERING UI vs RULES: preserve flow-emit order within a player's batch (commandQueues is FIFO; iterate in arrival order, append in order). A flow doing ui_clear then rule_install must serialize as [clear, rules_install] in the batch so the client applies them in order (ui_widgets/rules each consume their own subs in ipairs order). No cross-channel race because there is exactly ONE channel and ONE ordered envelope.
  - CLIENT SUB-COMMAND ROUTING: make the client fan-out route EACH sub-command by its OWN action prefix to RulesEngine (rules_/state_) vs UIWidgets (else) — NOT recurse blindly into UIWidgets. Put this single router in modmain's dstp_ui_dirty listener (or a shared dispatch it calls for both the top-level cmd and each batch sub). Add a 'batch' case to rules_engine.lua ONLY as a defensive no-op/iterator so a rules sub never hits 'unknown action'.
  - REMOVE DOUBLE FAN-OUT: delete/neutralize the internal UIWidgets 'batch' branch (ui_widgets.lua:1089-1095) once modmain owns fan-out, so each sub-command is processed exactly once. Guard against duplicate widget creation.
  - EMPTY/SINGLE BATCH: if a player ends with exactly one sub-command, the existing code re-issues a plain command (core.lua:258) — keep that fast path but it now applies to ALL families, and it must still carry the envelope seq for dedup. Decide: single-sub still wraps in a 1-element batch to carry seq uniformly, OR the single sub carries the seq itself. Pick one and make client dedup read seq from whichever wrapper is sent.
  - STALE-CLIENT GRACEFUL DEGRADE: a mixed batch delivered to an OLD client (no fan-out, old UIWidgets batch branch) must not crash — UI subs process, rules/state subs hit 'unknown action' and are logged+dropped (no worse than today). New backend + old client must never produce a hard error.
  - NO SEQ ON SUB-COMMANDS LEAK: ensure ui_command/ui_broadcast subs (which never had seq) do not accidentally inherit a per-command seq that would advance the shared _seq incorrectly; seq lives on the envelope exclusively.
