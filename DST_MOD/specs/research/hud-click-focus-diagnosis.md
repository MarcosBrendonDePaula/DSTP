# HUD click is focus-based — root-cause diagnosis (#16)

> Saved from workflow `Find the REAL reason DSTP overlay/text clicks fall through while the tree button works`. Raw multi-agent research output;
> see the sibling specs for the distilled conclusions.

## ground

**1.** 
  - **summary:** CONFIRMED. The user's hypothesis is correct. In vanilla DST, HUD click routing is FOCUS-based, not position-based. A left-click reaches a HUD widget only if that widget currently holds focus in the focus chain, and focus is granted by mouse position via the engine hit-test (which only includes entities flagged SetClickable(true)) on the per-frame hover update. A widget that never gains focus never receives OnControl, so the CONTROL_PRIMARY event is left unconsumed by the UI and falls through to the PlayerController (the world).

The full trace of a left-click:

1. The engine fires TWO Lua callbacks for one click: OnMouseButton(button, down, x, y) which carries the screen x,y, and OnControl(CONTROL_PRIMARY, down) which carries NO position. (input.lua: OnMouseButton 179, OnControl 163).

2. Routing is to the TOP SCREEN, not a hit-tested widget. FrontEnd:OnControl (frontend.lua:412) calls self.screenstack[#self.screenstack]:OnControl(...) (frontend.lua:430). FrontEnd:OnMouseButton (frontend.lua:1275) calls the top screen's OnMouseButton (frontend.lua:1282). Neither does any xy hit-test to pick a widget.

3. The screen walk is purely focus-chained. Screen does NOT override OnControl/OnMouseButton (widgets/screen.lua has only is_screen=true at line 10), so it inherits Widget:OnControl (widget.lua:124). That method: line 127 'if not self.focus then return false end', then line 129-131 recurses ONLY into children where v.focus is true. Widget:OnMouseButton (widget.lua:56) is identical: line 57 'if not self.focus then return false end', line 59 recurses only into v.focus children. So OnControl/OnMouseButton flow strictly DOWN the chain of focused widgets; an unfocused widget (and its subtree) is skipped entirely. The x,y in OnMouseButton is only passed to a leaf that ALREADY has focus — it does not select the widget.

4. How a widget GAINS focus = mouse position via hit-test. FrontEnd:DoHoverFocusUpdate (frontend.lua:918, called every frame at frontend.lua:866) does: entitiesundermouse = TheInput:GetAllEntitiesUnderMouse() (line 924); hover_inst = entitiesundermouse[1] (line 925); if hover_inst.widget then hover_inst.widget:SetFocus() (line 926-927) else the screen gets focus (line 929). The entity list comes from TheSim:GetEntitiesAtScreenPoint (input.lua:516/520, the C++ hit-test by screen point), which only includes entities flagged clickable via entity:SetClickable (Widget:SetClickable -> self.inst.entity:SetClickable, widget.lua:526-527). Widget:SetFocus (widget.lua:659) sets self.focus=true and propagates UP via parent:SetFocusFromChild (widget.lua:632), establishing the screen-to-leaf focus chain that step 3 walks.

5. Even the only natively-clickable HUD widget type (Button/ImageButton) gates its onclick on focus. Button:OnControl (widgets/button.lua:56) line 59: 'if not self:IsEnabled() or not self.focus then return false end'; the onclick fires only at line 84-85 and only when self.focus is set. A bare Text or Image overrides neither OnControl nor OnMouseButton (confirmed: no matches in widgets/image.lua or widgets/text.lua), so it always returns false from the inherited Widget:OnControl and can never consume a click.

6. Why the click passes through to the WORLD. Input:OnControl (input.lua:163) at line 166 only fires the world handler when 'not TheFrontEnd:OnControl(control, digitalvalue)' — i.e. when no focused UI widget consumed it — then runs self.oncontrol:HandleEvent(...) (input.lua:167-168). PlayerController subscribes to that via TheInput:AddGeneralControlHandler (playercontroller.lua:319 -> input.lua:144 AddControlHandler('oncontrol')), and its PlayerController:OnControl (playercontroller.lua:601) performs the world action/attack. So an unconsumed CONTROL_PRIMARY becomes a world click.

Net: routing is focus-based; position only feeds focus through the clickable-entity hit-test on hover. An ImageButton that is a CHILD of a non-clickable Text/Image whose entity is never returned by GetEntitiesAtScreenPoint will never have SetFocus called on its chain, so it never receives OnControl, returns nothing, and the click falls through to PlayerController/world. The user's mechanism is exactly right.

**2.** 
  - **summary:** CONFIRMED: the USER HYPOTHESIS is correct in mechanism but needs one correction about WHY. In DST the click pipeline is two-stage: (A) the ENGINE hit-test (`TheSim:GetEntitiesAtScreenPoint`) decides which entity is "under the mouse", and (B) `FrontEnd:DoHoverFocusUpdate` takes only the TOPMOST hit entity and calls `widget:SetFocus()` on it; then `OnControl` is dispatched down the tree but ONLY into widgets whose `.focus == true`. So a click fires a Button/ImageButton's onclick ONLY if that exact widget won the hover hit-test and thereby holds focus. A bare Text/Image never has an onclick handler so it is "not clickable" in the actionable sense — but importantly, the focus chain is NOT gated by the widget's PARENT being a Button: focus is awarded by the engine's per-entity clickable hit-region (z-order topmost), and `SetFocus` then propagates UP via `SetFocusFromChild`, lighting the whole ancestor chain. The real failure mode for an overlay that "passes through to the world" is that the overlay's clickable hit-region never wins `GetEntitiesAtScreenPoint` (no/empty region, draw-order behind, SetClickable(false), or a sibling drawn on top), so it never becomes `entitiesundermouse[1]`, never gets `SetFocus()`, and its `OnControl` returns at the `if not self.focus then return` guard. Whether the overlay's PARENT is a Text/Image is irrelevant to reachability — what matters is the overlay ITSELF winning the engine hit-test.
  - **facts:** 
    - FOCUS IS REQUIRED BEFORE A CLICK REGISTERS (Q1 = YES). ImageButton:OnControl bails immediately if not focused: imagebutton.lua:175 `if not self:IsEnabled() or not self.focus then return end`. Button:OnControl does the same: button.lua:59 `if not self:IsEnabled() or not self.focus then return false end`. And the base Widget:OnControl gate: widget.lua:127 `if not self.focus then return false end`.
    - FOCUS IS DRIVEN BY THE MOUSE-HOVER HIT-TEST, NOT BY TREE POSITION. frontend.lua:918-931 `FrontEnd:DoHoverFocusUpdate`: it calls `local entitiesundermouse = TheInput:GetAllEntitiesUnderMouse()` (line 924), takes ONLY the topmost `local hover_inst = entitiesundermouse[1]` (line 925), and `if hover_inst and hover_inst.widget then hover_inst.widget:SetFocus()` (lines 926-927). If nothing is hit it focuses the screen instead (lines 928-929). DoHoverFocusUpdate is called every frame from the update loop: frontend.lua:866 `self:DoHoverFocusUpdate()`.
    - THE 'ENTITIES UNDER MOUSE' COME FROM A NATIVE ENGINE HIT-TEST AGAINST EACH ENTITY'S CLICKABLE REGION. input.lua:521 `self.entitiesundermouse = TheSim:GetEntitiesAtScreenPoint(TheSim:GetPosition())` (also input.lua:516 in UpdateEntitiesUnderMouse). GetAllEntitiesUnderMouse just returns that list: input.lua:263-265 `return self.mouse_enabled and self.entitiesundermouse or {}`. So which widget gets focus is decided by the C++ side from each entity's render/clickable region and draw order (z), not by the Lua parent/child structure.
    - HIT REGION (Q2): an ImageButton's clickable area is its child Image's texture region, optionally resized. ImageButton ctor adds the Image: imagebutton.lua:7 `self.image = self:AddChild(Image())`. Default region = the texture's own pixel size (Image:SetTexture updates metrics: image.lua:91-94). ForceImageSize sets size_x/size_y and SCALES the image to that size: imagebutton.lua:37-41 `self.size_x=x; self.size_y=y; self.image:ScaleToSize(self.size_x,self.size_y)`. ScaleToSize is pure scale, NOT a new region: image.lua:127-132 computes scalex=w/w0 and calls SetScale. SetSize changes the actual ImageWidget region: image.lua:107-113. So a blank/empty or zero-size texture = no usable hit region = never wins the hit-test. The blank.tex hover-button pattern in widget.lua:759-760 explicitly ScaleToSize's a blank image to the parent's region size precisely to GIVE it a hit area.
    - SetClickable TOGGLES whether the entity participates in the hit-test at all. widget.lua:526-528 `function Widget:SetClickable(val) self.inst.entity:SetClickable(val) end`. Non-interactive decorations explicitly opt OUT: e.g. frontend.lua:78 `self.blackoverlay:SetClickable(false)` and hovertext bg/text widget.lua:730/734 `SetClickable(false)`. An entity with SetClickable(false) will not appear in GetEntitiesAtScreenPoint, so it can never become entitiesundermouse[1] and never gets focus.
    - PARENT REQUIREMENT (Q3): the parent does NOT need to be a Button, be focusable, or call SetFocus for the child to be reachable by the mouse. Focus is granted to whichever ENTITY wins the engine hit-test (frontend.lua:925-927), and SetFocus then propagates focus UPWARD to all ancestors automatically: widget.lua:659-688 Widget:SetFocus sets self.focus=true and calls `self.parent:SetFocusFromChild(self)` (line 679); SetFocusFromChild walks up the whole chain lighting each ancestor's focus and clearing focused siblings: widget.lua:632-657 (note line 654 `self.parent:SetFocusFromChild(self)`). So a Button nested under a plain Text/Image parent IS reachable — PROVIDED the Button's own image wins the hit-test. There is also focus_forward (widget.lua:661-665) which only REDIRECTS SetFocus to another widget; nothing requires the parent to forward focus for a normal mouse hover to reach a child.
    - DRAW-ORDER / MoveToFront (Q4): yes, z-order matters because only the TOPMOST hit entity (entitiesundermouse[1]) gets focus (frontend.lua:925). MoveToFront/MoveToBack control that order: widget.lua:64-70 (`self.inst.entity:MoveToFront()` / `MoveToBack()`). Notably ImageButton pushes its own image to the BACK on construction so the button's text/overlays draw above it: imagebutton.lua:8 `self.image:MoveToBack()`. If another widget is drawn on top of the overlay at the same screen point, that other widget becomes entitiesundermouse[1] and steals the focus/click.
    - WHAT Button:OnControl CHECKS BEFORE onclick (Q5), in order (button.lua:56-94): (1) delegate to base first — button.lua:57 `if Button._base.OnControl(self,control,down) then return true end` (lets a focused child consume it); (2) enabled AND focused — button.lua:59 `if not self:IsEnabled() or not self.focus then return false end`; (3) not in Selected state — button.lua:61 `if self:IsSelected() and not self.AllowOnControlWhenSelected then return false end`; (4) control matches and mouse-primary check — button.lua:63 `if control == self.control and (not self.mouseonly or TheFrontEnd.isprimary) then`. self.control defaults to CONTROL_ACCEPT (button.lua:25), and FrontEnd remaps a real left-click CONTROL_PRIMARY to CONTROL_ACCEPT before dispatch (frontend.lua:428 `:OnControl(control == CONTROL_PRIMARY and CONTROL_ACCEPT or control, down)`). onclick fires on the UP edge only when a prior down was recorded: button.lua:80-88 `else if self.down then self.down=false ... if self.onclick then self.onclick() end`. There is NO explicit hit-region test inside OnControl — the hit-region test already happened upstream in the engine hit-test that granted focus; OnControl only re-checks self.focus.
    - ImageButton:OnControl mirrors this (imagebutton.lua:174-223): same focus/enabled/selected guards (line 175, 177), control match via `control == TheInput:ResolveVirtualControls(self.control)` (line 179), down-edge sets self.down + image_down + ondown (lines 180-203), and the up-edge `else if self.down then ... if self.onclick then self.onclick() end` (lines 204-217). So onclick requires a matched down THEN up while continuously focused; losing focus mid-press cancels it — Button:OnLoseFocus clears self.down (button.lua:128-132).
    - CONCLUSION re: the overlay hypothesis. The mechanism the user describes (hover->focus->OnControl, only focused Button-type widgets consume clicks, bare Text/Image is not clickable) is exactly what the code shows. The ONE correction: a child Button is NOT blocked merely because its PARENT is a non-focusable Text/Image — focus propagates up from the child (widget.lua:679,654). The overlay 'passes through to the world' because the overlay's OWN entity never becomes entitiesundermouse[1] in the engine hit-test (frontend.lua:925, input.lua:521): most likely it has no real clickable region (blank/zero-size texture not ScaleToSize'd to a region, cf. widget.lua:759-760), or SetClickable was left false (widget.lua:526), or it is drawn behind/under something else (MoveToFront/z, widget.lua:68; imagebutton image MoveToBack at imagebutton.lua:8). Without winning the hit-test it never gets SetFocus(), so its OnControl returns at the `if not self.focus` guard (imagebutton.lua:175) and the input falls through to the world/playercontroller.
  - **uncertainties:** 
    - The actual point-in-region geometry test lives in the C++ engine (TheSim:GetEntitiesAtScreenPoint / entity:SetClickable / AddImageWidget hit-region) and is not visible in these Lua files. I can prove from Lua that focus follows entitiesundermouse[1] and that hit-region is governed by texture size + ScaleToSize/SetSize + SetClickable, but the exact pixel/alpha rules (e.g. whether fully-transparent pixels of a real texture still register, or whether it is a pure bounding-box test) are engine-side and not determinable from these files alone.
    - Exact draw-order/z-sorting rules between sibling widgets at the same screen point are engine-side. The Lua shows MoveToFront/MoveToBack (widget.lua:64-70) and that imagebutton MoveToBack's its image (imagebutton.lua:8), and that only entitiesundermouse[1] wins, but the precise ordering when multiple clickable entities overlap (insertion order vs explicit MoveToFront vs UITransform depth) is not fully specified in these files.
    - CanMouseThrough (input.lua:526-557) can skip a hovered entity in favor of the next one underneath, but that path applies to world-Transform entities; whether a HUD overlay could be affected depends on engine flags not shown here. Did not fully trace its interaction with pure-HUD widgets (Transform == nil).
    - The user's specific overlay widget code was not provided/inspected, so I cannot confirm WHICH of the three concrete causes (no region / SetClickable false / drawn behind) applies to their case — only that it must be one (or more) of these, since those are the only things in the Lua that gate winning the hit-test. Per instructions I did not inspect or propose their fix.
