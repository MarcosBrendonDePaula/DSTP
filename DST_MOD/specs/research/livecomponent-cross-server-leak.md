## findings

Live.use(LiveDSTP) (L971) passes no server_id, so the backend has no scoping signal. The same pattern applies to LiveAutomation (flows:/logs:/folders: per server in one singleton).

BACKEND (builds a multi-server delta):
- LiveDSTP.ts:31 `static singleton = true`, no room/server scoping. pushStoreUpdate() (L62-128) calls dstStateStore.getServerGroups() which (DSTStateStore.ts:106-140) iterates ALL shards across ALL servers, and writes flat keys for EVERY group: delta.serverIds = ALL ids (L73), delta[`server:${g.server_id}`] (L81: name/online/active_events/debounce/per-shard day/season/phase) and delta[`players:${g.server_id}`] (L108: full player map keyed by userid — position, health/hunger/sanity, inventory, admin, etc.) for every server. delta.events = dstStateStore.getAllEvents() (L120), which (DSTStateStore.ts:142-148) concatenates EVERY shard's events across ALL servers (chat_message contents, deaths, combat) and slices last 500 — one global array, no per-server split. Driven by a 5s setInterval (L54) + notifyLiveDSTP() (L25-27).

WIRE / BROADCAST (the SAME multi-server object goes to every client):
- setState(delta) → state manager _applyStateUpdate → emitFn("STATE_DELTA", { delta }) at node_modules/@fluxstack/live/dist/index.js:2717.
- For a singleton, emitFn is the EMIT_OVERRIDE installed at index.js:3899-3919: JSON.stringify(message) ONCE, then `for (const [cId, cWs] of singleton.connections) cWs.send(serialized)` — fans the identical serialized delta to EVERY connected client with ZERO per-connection/per-server filtering.
- On connect (index.js:3805-3825): a new client of the existing singleton is added to the shared `existing.connections` map (L3808) and IMMEDIATELY sent the FULL getSerializableState() via STATE_UPDATE (L3811-3819). getSerializableState() (index.js:2766-2768) returns the entire flat _proxyState — i.e. all servers' server:*/players:*/events at once, unfiltered.
- Live.use(LiveDSTP) (DSTPanel.tsx:971) passes NO server_id, so the backend has no scoping signal even in principle.

CLIENT (filter is display-only, AFTER the data is already on the wire):
- DSTPanel.tsx reads state[`server:${selectedServer}`] (L978), state[`players:${selectedServer}`] (L986), and allEvents.filter(e => e.server_id === selectedServer) (L991). The other servers' server:<id>/players:<id> keys and the merged global events[] are already present in `state` — a client viewing server A receives server B's data and merely doesn't render it. Visible via devtools / raw WS frames.

PARALLEL: LiveAutomation.ts:78 is also `static singleton = true`; syncState() (L156) pushes flat flows:<id>/logs:<id>/folders:<id> via the same this.setState → singleton-fanout, so flow definitions/logs/folders for ALL servers ride the same shared broadcast.

Impact: any authenticated panel WS client receives, on the wire, every server's player roster (incl. admin flags, positions, inventories, health/hunger/sanity), per-server config, and the global event stream (chat/deaths/combat) for servers it is not authorized to view. The per-server PanelAuth gate does not constrain what LiveDSTP/LiveAutomation broadcast — fix requires per-connection/per-server scoping (room-based emit or server-scoped Live Component), not a client filter.
  - **notes:** Real cross-tenant leak on the wire, not just a hidden-UI artifact. An admin authorized for server A who opens DevTools (or reads the WS frames) sees server B's full state: player positions, health/hunger/sanity, inventories, userids, admin flags, AND all of server B's chat messages and events (getAllEvents is global). The per-server auth model (PanelAuthStore, authorized per server_id) is therefore NOT enforced on the data plane — auth gates the page/actions, but the singleton WebSocket state is shared verbatim. Same structural issue in LiveAutomation (flows:/logs:/folders: for all servers in one singleton) — flow definitions and logs of other servers also go out on the wire. Note: the framework DOES support room scoping (rooms partition broadcasts via broadcastToRoom), but LiveDSTP/LiveAutomation use plain `singleton = true` with no room, so all clients share one connection set. Proper fix would be per-server rooms (subscribe each connection to a `server:<id>` room and only emit that server's keys to it) or splitting into per-server component instances keyed by server_id, rather than one global singleton holding all tenants' data.

(1) Both components are singletons: LiveAutomation.ts:78 `static singleton = true`; LiveDSTP.ts:31 `static singleton = true`.

(2) Keys are server-scoped by NAME only, in ONE shared state object. LiveAutomation.ts:156 syncState => setState({ `flows:${serverId}`, `logs:${serverId}`, `folders:${serverId}` }); clearLogs at :329 sets `logs:${serverId}`. LiveDSTP.ts:62-128 pushStoreUpdate builds a SINGLE delta containing `serverIds` (ALL ids), and for EVERY group `server:${id}` (:77/81) + `players:${id}` (:78/108), plus one global `events` (:120). The source spans all servers: DSTStateStore.getServerGroups() (:106-109) iterates ALL shards across ALL servers; getAllEvents() (:142-147) concatenates events from every shard globally. So one LiveDSTP delta legitimately contains server A's AND server B's data.

(3) The fan-out has ZERO transport isolation. setState (node_modules/@fluxstack/live/dist/index.js:2674) computes actualChanges via deepDiff and calls emitFn('STATE_DELTA',{delta:actualChanges}) at :2717. For a singleton, emitFn resolves to the EMIT_OVERRIDE_KEY closure (index.js:3899-3929, wired via getEmitOverride at :3441). That closure serializes ONCE and loops `for (const [cId, cWs] of singleton.connections) cWs.send(serialized)` (:3911-3913) — every connected admin WebSocket gets the IDENTICAL message. No per-connection, per-server, per-room, or auth filtering on the send path.

(4) Connections accumulate across servers: when a SECOND admin opens the panel (any server), mountComponent finds the existing singleton and does `existing.connections.set(connId, ws)` (index.js:3805-3808) — joining the same connections map that the fan-out iterates. So an admin viewing only server B is in the same connections set that receives server A's deltas.

(5) Worst case — new client gets the FULL accumulated multi-server state immediately: on join, index.js:3811-3819 sends `existing.instance.getSerializableState()` (the entire singleton state with server:A, players:A, flows:A, logs:A, capture:A already accumulated) to the new ws via STATE_UPDATE, before any delta. So opening server B's panel synchronously delivers every other server's data.

(6) Worker-core capture traces reach the same singleton: ServerCoreManager.ts:81-82 `case 'emitState': panelEmit?.(args[0])` -> setPanelEmitter seam -> LiveAutomation.ts:19-21 `_automationInstance.setState(delta)` -> same singleton fan-out. So per-server worker capture (`capture:${serverId}`) also broadcasts to all panels.

(7) The client-side filter is DISPLAY-ONLY and does not gate the wire: AutomationPage.tsx:57/59/60 read `(auto.$state as any)['flows:'+urlServer]` / `logs:` / `capture:`; DSTPanel.tsx:978/986/990 read `state['server:'+selectedServer]` / `players:` / `state.events`. These are key lookups into the ONE shared $state object that already holds every server's keys in client memory — the unselected servers' data is fully present in the WS frames and the JS heap; the UI merely doesn't render it. Anyone with the panel open (authorized for any one server) can read other servers' flows/logs/players/events/capture-traces via DevTools or the WS frames.

This is NOT display-only-safe and NOT fully-isolated. It is a genuine leak-on-wire: data for servers a given admin did not select (and, depending on the auth model, may not be authorized for) is transmitted verbatim to that admin's browser. whatLeaks = flows:<serverId>, logs:<serverId>, folders:<serverId>, capture:<serverId> (LiveAutomation) and additionally server:<serverId>, players:<serverId>, the global events[] (LiveDSTP) — the claim under-counts LiveDSTP's leaked keys but is correct on every mechanism.

Caveat on real-world impact: this is only a cross-TENANT leak if a single backend hosts multiple servers owned by DIFFERENT untrusted admins AND PanelAuthStore authorizes a session per-server (so server-A admin shouldn't see server-B). Per CLAUDE.md the panel auth IS per-server (`authorized per server_id`), so the leak is meaningful. If in practice every panel admin is authorized for all servers on the box, the wire-leak still exists but the trust impact is lower. The transport defect itself is real and confirmed regardless.

RUNTIME FANS THE SAME STATE TO EVERY CLIENT (no per-client/room filtering) — verified in node_modules/@fluxstack/live/dist/index.js:
- On connect to an existing singleton (3805-3825): existing.connections.set(connId, ws) then sendImmediate(ws, STATE_UPDATE { state: getSerializableState() }) — the FULL current state (all servers) is pushed to the newly joined client immediately.

ComponentMessaging.emit (2828-2844) checks getEmitOverride() and, if set, calls override(type,payload) and RETURNS (short-circuits any room logic). For singletons the override IS set (3899-3919): it iterates singleton.connections and cWs.send(serialized) the IDENTICAL payload to EVERY connection — zero per-client filtering, no room scoping.

NO ROOM ISOLATION CONFIGURED: the framework supports rooms (CounterDemo/AuthDemo/RoomChatDemo pass room:), but DSTPanel.tsx:971 Live.use(LiveDSTP, {initialState}) and AutomationPage.tsx:14-15 Live.use(LiveAutomation,{...}) / Live.use(LiveDSTP) pass NO room → all clients share one singleton bucket + one connections map + one state object.

CONCLUSION: A client viewing server A receives server B's full server info, player objects (incl. admin flags, positions, inventories), global event/chat stream, and any flows/logs/folders/capture for every server present in the shared singleton state — all delivered over the WebSocket (STATE_UPDATE on join + STATE_DELTA on change) and visible in raw WS frames / devtools, regardless of the UI hiding it. The claimed whatLeaks list is accurate. Caveat: flows:/logs:/folders:<id> only populate when some action triggered syncState for that server (e.g. an AutomationPage was opened or a flow saved) — not strictly 'any client ever opened', but the claim's framing is essentially correct and the player/server/events leak is unconditional via the 5s timer.

The full multi-server state object lives in every connected client's received state and is plainly visible in React/devtools and raw WebSocket STATE_UPDATE/STATE_DELTA frames. LandingPage.tsx:698-699 deliberately avoids leaking the server LIST on the not-found page, but that cosmetic care is moot because serverIds + every server:/players:/flows: key is already in the same state payload the client received. Caveat on scope: server:<id>/players:<id>/events for a server appear as soon as that DST server syncs (always present for all online servers); flows:/logs:/folders:/capture:<id> only get written into the shared state when some connected admin's AutomationPage calls loadFlows/syncState for that server (LiveAutomation.ts:151) — but once written they persist in the singleton state and fan to everyone. Real-world impact depends on whether one backend hosts multiple DST servers with distinct admins (the documented 'one backend, many servers' model) — in that topology, admin A reading the WS frames sees admin B's players, inventories, admin grants, chat and automation flows. Fix direction: scope these singletons per server (room = serverId) or split state per connection so each client only receives its selected server's keys, rather than relying on client-side key selection.
