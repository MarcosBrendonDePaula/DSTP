## ground

**1.** 
  - **summary:** Full contract for DST's widgets/textedit.lua (TextEdit), focused on what it takes to make a freshly-created TextEdit on the HUD (PlayerHud.controls, not a modal Screen) accept typed characters and fire a callback on Enter. The whole HUD-vs-Screen problem is solved by ONE engine mechanism the file uses: SetForceEdit(true) + SetEditing(true), which routes ALL keystrokes/text input to this widget via TheFrontEnd:SetForceProcessTextInput. File read in full from /tmp/dstscripts/scripts/widgets/textedit.lua (lines 1-619); input dispatch confirmed in /tmp/dstscripts/scripts/frontend.lua.
  - **facts:** 
    - CONSTRUCTOR: TextEdit(font, size, text, colour) — textedit.lua:10. It is a subclass of Text (textedit.lua:10, `Class(Text, ...)`). Side effect: at textedit.lua:13 it calls `self.inst.entity:AddTextEditWidget()` which attaches the NATIVE engine widget exposed as `self.inst.TextEditWidget` (the C++ side that actually stores the edited string + cursor). It also calls `self:SetString(text)` (textedit.lua:14) and `self:SetEditing(false)` (textedit.lua:21). Defaults set in ctor: limit=nil, allow_newline=false (textedit.lua:19), enable_accept_control=true (textedit.lua:20), force_edit=false (textedit.lua:23).
    - HOW IT CAPTURES THE KEYBOARD — THE KEY TO THE HUD PROBLEM: SetEditing(true) (textedit.lua:92). When transitioning false->true (textedit.lua:95-117) it: sets self.editing=true; calls self:SetFocus() (textedit.lua:99); DoSelectedImage() (highlight, :101); TheInput:EnableDebugToggle(false) (:102, so typed keys don't trigger debug); sets edit text colour (:111-113); and CRUCIALLY `if self.force_edit then TheFrontEnd:SetForceProcessTextInput(true, self) end` (textedit.lua:115-117). It also calls self.inst.TextWidget:ShowEditCursor(true) (textedit.lua:145).
    - SetForceProcessTextInput is the entire reason this works on the HUD. frontend.lua:1169 — `FrontEnd:SetForceProcessTextInput(takeText, widget)` stashes the widget as self.textProcessorWidget and sets self.forceProcessText=true (frontend.lua:1175-1176). Then FrontEnd's GLOBAL input handlers route EVERYTHING to that widget, BYPASSING the active screen and the player's WASD/hotkey controls: FrontEnd:OnRawKey (frontend.lua:1186) does `if self.forceProcessText and self.textProcessorWidget ~= nil then self.textProcessorWidget:OnRawKey(key, down)` (frontend.lua:1193-1194); FrontEnd:OnTextInput (frontend.lua:1201) does `... then self.textProcessorWidget:OnTextInput(text)` (frontend.lua:1208-1209). These FrontEnd handlers are registered ONCE globally via TheInput:AddKeyHandler / TheInput:AddTextInputHandler (frontend.lua:213-214), so they fire regardless of whether the widget lives on the HUD or on a pushed Screen. This is exactly why a HUD-resident TextEdit can grab the keyboard without being on a modal Screen — you do NOT need to push a Screen.
    - SetForceEdit(force) — textedit.lua:71-73: just sets self.force_edit. It must be true BEFORE SetEditing(true) for the SetForceProcessTextInput branch (textedit.lua:115) to fire. force_edit defaults to false (textedit.lua:23), so a plain SetEditing(true) on a HUD widget gives focus but does NOT capture the global keyboard — input would still go to the screen/player. So on the HUD you MUST call SetForceEdit(true).
    - TYPED CHARACTERS land in OnTextInput(text) (textedit.lua:206), text is ONE char. It bails if not (self.shown and self.editing) (textedit.lua:211) — so the widget must be shown AND editing. It applies conversions (textedit.lua:220), ValidateChar (textedit.lua:222), then forwards to the native widget: `self.inst.TextEditWidget:OnTextInput(text)` (textedit.lua:226) which actually inserts the char. Returns true (or true,overflow). There is NO per-keystroke user callback for content from OnTextInput itself — the live-typing hook is OnTextInputted (see below).
    - NON-TEXT KEYS (backspace, arrows, enter, tab, paste) go through OnRawKey(key, down) (textedit.lua:267). While editing & down: paste key -> reads clipboard and feeds chars via OnTextInput (textedit.lua:280-289); allow_newline+ENTER -> inserts "\n" (textedit.lua:290-291); otherwise `self.inst.TextEditWidget:OnKeyDown(key)` (textedit.lua:293) handles backspace/cursor. It records editing_enter_down on ENTER (textedit.lua:295). On key UP, if key==KEY_ENTER and not self.focus and editing_enter_down and not allow_newline -> calls self:OnProcess() (textedit.lua:296-302). Up of other keys -> TextEditWidget:OnKeyUp (textedit.lua:318). It returns `not self.validrawkeys[key]` (textedit.lua:327) to GOBBLE keys so they don't reach debug/game.
    - ENTER / COMMIT path = OnProcess() (textedit.lua:243-249): calls self:SetEditing(false) (which un-captures input — textedit.lua:244), TheInputProxy:FlushInput() (textedit.lua:245), then `if self.OnTextEntered then self.OnTextEntered(self:GetString()) end` (textedit.lua:246-248). OnTextEntered is a plain field you assign (NOT a method, no default); it is called with the final string. There are TWO routes into OnProcess: (a) keyboard ENTER up via OnRawKey (textedit.lua:301, the common HUD/mouse case), and (b) CONTROL_ACCEPT via OnControl when already editing (textedit.lua:360-368). So to get an Enter callback you set `textedit.OnTextEntered = function(str) ... end`.
    - OnTextInputted(down) (textedit.lua:321-323) is the OPTIONAL live-typing callback: if assigned, OnRawKey calls `self.OnTextInputted(down)` after every raw key while editing. Receives the key down/up boolean (NOT the text); you read current text via GetString(). Use this to react to each keystroke (e.g. live validation/echo). It is nil by default.
    - ValidateChar(text) (textedit.lua:154-163): rejects control chars string.char(8,22,27) always, and 10/13 (newline/CR) unless allow_newline; honors self.validchars (whitelist) and self.invalidchars (blacklist). Configured via SetCharacterFilter(validchars) (textedit.lua:488) and SetInvalidCharacterFilter(invalidchars) (textedit.lua:492).
    - OTHER PUBLIC API: SetString(str) (textedit.lua:75) sets text (formats + pushes to native widget). GetString() is inherited from Text (returns displayed string; for passwords use GetLineEditString() textedit.lua:498 which returns the intended string). SetTextLengthLimit(limit) (textedit.lua:480) sets self.limit, enforced in OnTextInput (textedit.lua:212). SetAllowNewline(allow) (textedit.lua:83) — also recomputes enable_accept_control = not(allow_newline and editing) (textedit.lua:89) so that while editing with newlines allowed, ENTER inserts a newline instead of committing. SetPassword(to) (textedit.lua:502) -> native TextEditWidget:SetPassword. SetForceUpperCase (textedit.lua:506). SetTextPrompt(prompt,colour) (textedit.lua:602) ghost placeholder. SetTextConversion(in,out) (textedit.lua:200). EnableWordPrediction (textedit.lua:558). Disable() (textedit.lua:592) forces SetEditing(false).
    - HOW EDITING ENDS (3 ways): (1) ENTER -> OnProcess -> SetEditing(false) + OnTextEntered (textedit.lua:243-248 via :301 or :366). (2) CONTROL_CANCEL (Esc/B) while editing -> OnControl calls SetEditing(false) (textedit.lua:355-357), NO OnTextEntered fired. (3) Losing the force-input grab: if another widget calls SetForceProcessTextInput, FrontEnd calls this widget's OnStopForceProcessTextInput (frontend.lua:1173/1179) -> textedit.lua:257-265 -> SetEditing(false) + optional self.OnStopForceEdit(self) callback. SetEditing(false) (textedit.lua:118-140) reverses everything: clears editing, restores idle image/colour, and `if self.force_edit then TheFrontEnd:SetForceProcessTextInput(false, self)` (textedit.lua:132-134) RELEASES the global keyboard so WASD/hotkeys work again, and TheInput:EnableDebugToggle(true) (textedit.lua:139).
    - MINIMAL SET OF CALLS to make a freshly-created HUD TextEdit accept typed chars + fire on Enter: (1) `local te = parent:AddChild(TextEdit(font, size, "", colour))` — constructor (textedit.lua:10), font e.g. CHATFONT/DEFAULTFONT, size in px. (2) `te:SetForceEdit(true)` (textedit.lua:71) — REQUIRED on the HUD so SetEditing routes input through TheFrontEnd:SetForceProcessTextInput (textedit.lua:115). (3) `te.OnTextEntered = function(str) ... end` — assign the Enter callback field consumed by OnProcess (textedit.lua:246). (4) `te:SetEditing(true)` (textedit.lua:92) — gives focus AND, because force_edit is set, calls SetForceProcessTextInput(true,self) so the global FrontEnd key/text handlers (frontend.lua:1193-1194, 1208-1209) feed THIS widget instead of the player movement/hotkeys. After that, typed chars insert via OnTextInput->native widget, and pressing ENTER fires OnTextEntered(GetString()). The widget must be shown (OnTextInput guard at textedit.lua:211). Optional but recommended: SetTextLengthLimit, SetCharacterFilter, SetEditCursorColour (default set in ctor :43), SetTextPrompt for placeholder.
    - WHY this beats the existing 'ImageButton overlay + SetClickable + SetFocus' HUD-click trick for TEXT: focus + clickability only routes mouse/CONTROL events; raw typed keys are dispatched GLOBALLY by FrontEnd, and without forceProcessText they go to the active screen and then to player controls (WASD = movement, letters = hotkeys). SetForceEdit(true)+SetEditing(true) is the ONLY documented path in this file that makes FrontEnd send OnRawKey/OnTextInput to an off-screen-graph (HUD) widget (frontend.lua:1193-1209). To start editing on a click, your overlay's click handler should call te:SetEditing(true) (note: TextEdit:OnMouseButton at textedit.lua:149 is intentionally a no-op, and the built-in start-editing trigger is CONTROL_ACCEPT in OnControl at textedit.lua:360-363, which the HUD overlay click can emulate by directly calling SetEditing(true)).

In normal gameplay the HUD itself is the active screen, so this holds — but if some other modal screen is pushed on top, forceProcessText still routes to OUR textProcessorWidget (the check is on forceProcessText, not on which screen owns the widget), which is what we want but means our HUD field can keep capturing keys even when another screen is open. Behavior when truly no screen is active (loading) is untested.
    - GetString() is inherited from Text (not shown in this file); I confirmed the contract from usage (textedit.lua:247, 211) but did not open widgets/text.lua to verify it returns the live edited string vs a cached one. For password fields the file explicitly says to use GetLineEditString() (textedit.lua:496-500), implying GetString() may be masked — worth confirming if passwords are ever used.
    - The native `self.inst.entity:AddTextEditWidget()` and `self.inst.TextEditWidget` (OnTextInput/OnKeyDown/OnKeyUp/SetString/SetPassword) are C++ engine calls; their exact semantics (e.g. UTF-8 handling, cursor) are not in Lua and were inferred from call sites.
    - editing_enter_down logic (textedit.lua:295-302) commits on ENTER-UP only when `not self.focus`. On the HUD our widget normally HAS focus, so the OnRawKey ENTER-up branch at :296 (`key==KEY_ENTER and not self.focus`) may NOT trigger OnProcess; in the focused case the commit likely comes through OnControl CONTROL_ACCEPT (textedit.lua:360-368) instead. **2.** 
  - **summary:** YES — on the bare HUD (no modal Screen), `self.chat_edit:SetForceEdit(true)` plus `SetEditing(true)` (which calls `TheFrontEnd:SetForceProcessTextInput(true, self)`) IS sufficient to (a) route typed characters into the TextEdit and (b) stop WASD/hotkeys from reaching the player. A modal screen is NOT required. The single load-bearing global flag is `TheFrontEnd.textProcessorWidget`: setting it via force-process simultaneously reroutes keyboard input AND makes the HUD report input focus, which disables the player controller. The one prerequisite is that there must be an active FrontEnd screen — and there always is, because the PlayerHud itself is pushed onto the FrontEnd as a Screen (player_common.lua:742). For the DSTP `text_input` HUD widget, you must: create a `widgets/textedit` child, call `:SetForceEdit(true)`, and on click call `:SetEditing(true)` (mirroring chatinputscreen). The TextEdit's own `OnRawKey` gobbles unregistered keys (textedit.lua:327) so they don't leak to debug/gameplay, and `OnStopForceProcessTextInput`/`OnStopForceEdit` give you the blur/commit hook to send the typed value back to the backend.
  - **facts:** 
    - THE MECHANISM (full chain, all Lua-verified): TextEdit:SetEditing(true) -> SetForceProcessTextInput(true, self) -> TheFrontEnd.textProcessorWidget=self -> PlayerHud:HasInputFocus()==true -> PlayerController:IsEnabled()==false -> WASD movement skipped. Simultaneously FrontEnd:OnRawKey/OnTextInput route keys to the widget instead of the screen/gameplay.
    - textedit.lua:115-117 (SetEditing): `if self.force_edit then TheFrontEnd:SetForceProcessTextInput(true, self) end`. This is the ONLY thing that wires a TextEdit into the force-process keyboard path. It only fires when force_edit is set (via SetForceEdit(true), textedit.lua:71-73). The mirror at textedit.lua:132-134 turns it off when editing stops.
    - frontend.lua:1169-1184 (SetForceProcessTextInput): when takeText and widget, it stops the previous textProcessorWidget, then sets `self.textProcessorWidget = widget; self.forceProcessText = true`. Passing widget==nil (or the same widget) with takeText=false clears it. This is the single global flag everything keys off of.
    - frontend.lua:1186-1199 (OnRawKey) and 1201-1213 (OnTextInput): KEY ROUTING. `local screen = self:GetActiveScreen(); if screen ~= nil then if self.forceProcessText and self.textProcessorWidget ~= nil then self.textProcessorWidget:OnRawKey/OnTextInput(...) else screen:OnRawKey/OnTextInput(...) end end`. When force-process is on, typing is delivered DIRECTLY to the widget, bypassing the screen's focus chain entirely. CAVEAT: requires GetActiveScreen() ~= nil.
    - WHY A MODAL ISN'T NEEDED — playerhud.lua:1081-1100 (PlayerHud:HasInputFocus): returns true when `TheFrontEnd.textProcessorWidget ~= nil` (line 1088), independent of any modal screen. So merely setting force-process makes the HUD claim input focus.
    - playercontroller.lua:491-502 (IsEnabled): `if ... not iscontrollerenabled then return false elseif self.inst.HUD ~= nil and self.inst.HUD:HasInputFocus() then return false, (...gameplay-allow exceptions...) end; return true`. Because HasInputFocus() is true while text-editing, IsEnabled() returns false, so OnUpdate's movement/DoDirectWalking branch is skipped (playercontroller.lua:2652, 2684-2696). The gameplay-allow exception at line 496 explicitly ANDs `TheFrontEnd.textProcessorWidget == nil`, i.e. text-editing is never treated as gameplay-allowing.
    - MOVEMENT IS ENGINE-POLLED, NOT EVENT-DRIVEN: playercontroller.lua:22-23 reads `TheInput:GetAnalogControlValue(CONTROL_MOVE_*)`; input.lua:479-487 IsControlPressed/GetAnalogControlValue read `TheSim:GetDigitalControl`/`GetAnalogControl` (C++) directly and are NOT gated by forceProcessText. So WASD is suppressed ONLY because IsEnabled() short-circuits the controller update — NOT because the engine zeroes the keys. This is why the textProcessorWidget->HasInputFocus->IsEnabled chain is what matters.
    - KEYS DON'T LEAK TO DEBUG/HOTKEYS: textedit.lua:267-328 (TextEdit:OnRawKey) ends with `return not self.validrawkeys[key]` (line 327) — it gobbles every unregistered raw key so it never reaches DoDebugKey/gameplay. frontend.lua:1194 calls the widget's OnRawKey instead of the `screen:OnRawKey ... DoDebugKey` branch, so debug keys are bypassed entirely while editing.
    - THE HUD IS A FRONTEND SCREEN (satisfies the GetActiveScreen() ~= nil requirement): player_common.lua:741-742 `local hud = PlayerHud(); TheFrontEnd:PushScreen(hud)`, and playerhud.lua:1 `local Screen = require "widgets/screen"`. With no modal open, the HUD is the active screen, so FrontEnd:OnRawKey/OnTextInput's `screen ~= nil` guard passes and routing to textProcessorWidget works.
    - WHAT chatinputscreen DOES (the reference, but it IS a modal): chatinputscreen.lua:25-30 OnBecomeActive -> `self.chat_edit:SetFocus(); self.chat_edit:SetEditing(true)`; chatinputscreen.lua:258 `self.chat_edit:SetForceEdit(true)`; chatinputscreen.lua:259 sets OnStopForceEdit to close. For the HUD you replicate the SetForceEdit(true)+SetEditing(true) calls but do NOT need the surrounding Screen — trigger SetEditing(true) from your existing ImageButton click handler.
    - COMMIT/BLUR HOOK: textedit.lua:257-265 (OnStopForceProcessTextInput) calls SetEditing(false) then self.OnStopForceEdit(self); textedit.lua:243-249 (OnProcess) runs on Enter, calls SetEditing(false), TheInputProxy:FlushInput(), then self.OnTextEntered(self:GetString()). Use OnTextEntered (enter/commit) and OnStopForceEdit (blur) to push the typed value to the DSTP backend. textedit.lua:206-241 OnTextInput is where chars actually land; it no-ops unless `self.shown and self.editing`.
    - WHY force_edit (not just focus) is required on the HUD: widget.lua Widget:OnRawKey/OnTextInput only descend into children that have `.focus` AND require `self.focus` (returns false otherwise). Without force-process, a HUD TextEdit would have to be in the focus chain to get keys, and raw keys would ALSO still drive gameplay/debug. Force-process delivers keys directly to the widget regardless of the focus chain and flips HasInputFocus to kill gameplay — both in one flag.
    - BONUS suppression while editing: textedit.lua:102 and 139 toggle TheInput:EnableDebugToggle(false/true) on edit start/stop; OnControl (textedit.lua:338-372) gobbles non-CANCEL/ACCEPT controls while editing (line 351-352); OnFocusMove returns true while editing (line 379) so WASD-as-focus-move is also eaten.

**3.** 
  - **summary:** A `text_input` node fits cleanly into the existing tree renderer with NO new RPC/event/trigger plumbing. It slots as a new `elseif t == "text_input"` branch in RenderNode (ui_widgets.lua), built from the engine's `widgets/textedit.lua`. The hard part — typed chars going INTO the field instead of WASD/hotkeys — is solved by TextEdit's own SetEditing/OnRawKey focus capture, the SAME focus principle the existing MakeHitTarget pattern relies on. The typed VALUE reaches the backend by REUSING the ui_callback path, but the in-tree callback signature is currently `ctx.callback_fn(cb, root_id)` and does NOT carry a payload today — the RPC and event layers already support a 3rd `data_json` arg (proven by rules_engine), so the fix is to (a) widen ctx.callback_fn to pass a 3rd value and (b) widen modmain's SetCallbackHandler closure to forward it instead of the hardcoded `""`. Backend SET of the field value works today via the existing Register/SetProps mechanism (add a `text`/`value` prop to the node's patch closure); backend READ-BACK does not exist as a pull — the field must PUSH its value via ui_callback (e.g. on Enter), since byId is a write-only patch registry with no value read-out.
  - **facts:** 
    - WHERE THE BRANCH SLOTS IN: RenderNode is one long if/elseif dispatch on `node.type` at ui_widgets.lua:431-714 (col/row :439, tabs :444, text :512, icon :540, image :560, button :574, bar :600, spacer :636, panel :639, unknown fallback :712). A `text_input` branch is a new `elseif t == "text_input" then ... return widget, w, h` inserted alongside these (most naturally after `button` at :598, since it is an interactive primitive). Like every branch it must return `widget, w, h` so LayoutChildren (:248) can stack it.
    - REGISTRATION/DISPATCH: text_input is a TREE node type, so it needs NO new entry in the CREATORS (:918) / UPDATERS (:928) flat dispatch tables — those are only for the legacy flat widgets (notification/label/panel/button/progress_bar/tree/follow). A tree node is reached only through CreateTree (:718) → RenderNode. The flat FlatAdapter path (:113) is irrelevant here.
    - CALLBACK SIGNATURE TODAY CARRIES NO PAYLOAD: in-tree clickables call `ctx.callback_fn(cb, ctx.root_id)` — exactly two args (MaybeClickable :366, tree button :592). ctx is built at CreateTree :722 as `{ callback_fn = UIWidgets._callback_fn, root_id = cmd.group or cmd.id, byId = {} }`. So firing `ctx.callback_fn(callback, value)` on Enter would put the typed value in the `root_id` slot — WRONG. A 3rd parameter must be added end-to-end to carry a value.
    - THE PAYLOAD CHANNEL ALREADY EXISTS ONE LAYER DOWN: UIWidgets._callback_fn is set by modmain's SetCallbackHandler closure (modmain.lua:158-165). That closure is `function(callback_name, widget_id)` and calls `SendModRPCToServer(MOD_RPC[modname]["UICallback"], callback_name, widget_id, "")` (:160) — note the hardcoded empty-string 4th arg (`data_json`).
    - THE RPC HANDLER ALREADY DECODES A PAYLOAD: AddModRPCHandler(modname, "UICallback", function(player, callback_name, widget_id, data_json) (modmain.lua:45). When data_json is non-empty it `json.decode`s it (:49-52) and attaches it to the event as `callback_data` (:60). The event pushed is `PushEvent("ui_callback", { userid, name, callback, callback_name, widget_id, callback_data })` (:54-61). So the backend ui_callback trigger ALREADY surfaces `{{trigger.callback_data}}` — the typed text just needs to ride that field.
    - PROOF THE PAYLOAD PATH WORKS: rules_engine.lua ActionEmitEvent (:220-241) already uses the same RPC with a real 4th arg — `SendModRPCToServer(MOD_RPC[_modname]["UICallback"], "rule_event:"..name, "", json_data)` (:235-240). So sending a JSON payload over UICallback is an established, working pattern; text_input would do the same but from the click/Enter handler instead of a rule.
    - EXACT 2-LINE WIRING TO EXPOSE THE TYPED VALUE (reuse, no new RPC): (1) widen the modmain closure to `function(callback_name, widget_id, data_json)` and forward `data_json or ""` instead of the literal "" at modmain.lua:160; (2) have the text_input branch fire `ctx.callback_fn(node.callback, node.id, json.encode({ value = <textedit:GetString()> }))` on submit. Then the backend reads the value at `{{trigger.callback_data.value}}`. The existing button/text/icon callers keep passing 2 args (3rd is nil → forwarded as ""), so they are unaffected.
    - BACKEND-SET (write) OF THE FIELD VALUE = EXISTING Register/SetProps MECHANISM: every addressable node calls Register(ctx, node, widget, patchClosure) (:297) which stores `ctx.byId[node.id] = { widget, patch }`; UIWidgets.SetProps (:733-741) looks up `entry.byId[cmd.node]` and calls `target.patch(props)`. The text_input branch just needs a patch closure that handles e.g. `if props.text ~= nil then textedit:SetString(tostring(props.text)) end` (mirrors the text node patch at :533-537). Then the backend pushes a value with the generic `ui_set`/`set` action (ProcessCommand :1068 → SetProps) — same path used for live HP bars and labels. `visible` is already handled generically inside Register (:301-305).
    - BACKEND-READ (pull) DOES NOT EXIST — value must be PUSHED: byId stores only a write-only `patch` closure (:299-307); there is no getter. The outward client→backend channel is solely the ui_callback event. UIWidgets exposes nothing that returns a widget's current value (only GetActiveCount :1084). So the design must be: field PUSHES its value via ui_callback on a trigger (Enter / a submit button / focus-loss), not the backend polling it. This matches the rest of the system (state flows up only via events).
    - THE WASD/HOTKEY PROBLEM AND ITS SOLUTION DIRECTION: the existing MakeHitTarget (:336-356) + MaybeClickable (:358-373) doc block (:310-335) explains HUD interactivity is FOCUS-based — only a focused widget gets OnControl, and an opaque `square.tex` ImageButton is used to win the engine hit-test so a HUD primitive becomes clickable WITHOUT stealing keyboard. For text_input the inverse is needed: while the field is FOCUSED/editing, keystrokes must be consumed by the TextEdit rather than reaching the player controller. DST's `widgets/textedit.lua` is built for exactly this: it overrides OnRawKey/OnTextInput and, when editing (SetEditing/`editing` flag, typically toggled by OnControl(CONTROL_ACCEPT)), returns true to swallow keys so they don't fall through to movement/hotkeys. So the branch should AddChild a TextEdit, give it a focusable hit region (same opaque-square trick as MakeHitTarget to gain focus on the HUD, since a bare TextEdit on PlayerHud.controls has the same non-focusable problem as Text), enter editing on click, and exit on Enter/Escape — calling the callback with GetString() on submit.
    - CreateTree wrapping detail: text_input is reached only via a tree (it is not one of the flat CREATORS). A flow author builds it inside a `tree`/`panel` UI like any other node; its `id` makes it addressable (SetProps) and its `callback` makes it report. The whole tree is anchor-positioned by CreateTree at :724-725; the text_input branch only returns its own (w,h).

- **hudCaptureMechanism:** No modal Screen needed. The PlayerHud IS itself a FrontEnd Screen (player_common.lua:742 PushScreen(hud)), so FrontEnd:OnRawKey/OnTextInput's `GetActiveScreen() ~= nil` guard is always satisfied during normal gameplay. The single load-bearing mechanism is TextEdit's force-process path: build a `widgets/textedit` child, call `te:SetForceEdit(true)` ONCE at creation, then call `te:SetEditing(true)` to begin editing. SetEditing(true) with force_edit set runs `TheFrontEnd:SetForceProcessTextInput(true, self)` (textedit.lua:115), which sets the global `TheFrontEnd.textProcessorWidget = te`. That one flag does BOTH halves of the problem: (a) FrontEnd:OnRawKey/OnTextInput now route every keystroke directly to `te` instead of the screen focus chain / player controller (frontend.lua:1193-1194, 1208-1209), and (b) PlayerHud:HasInputFocus() returns true (playerhud.lua:1088), so PlayerController:IsEnabled() returns false (playercontroller.lua:496) and the WASD/DoDirectWalking branch in OnUpdate is skipped — movement is engine-polled, suppressed only via this IsEnabled short-circuit, not by zeroing keys. TextEdit:OnRawKey also returns `not validrawkeys[key]` (textedit.lua:327) so unhandled keys are gobbled and never reach debug/hotkeys. A plain Text/ImageButton focus (the MakeHitTarget trick) is NOT enough for text: it only routes mouse/CONTROL events; raw typed keys are dispatched globally by FrontEnd and would drive movement/hotkeys unless force-process is on. So SetForceEdit(true)+SetEditing(true) is the ONLY path, and it is the same mechanism chatinputscreen uses minus the surrounding modal. Editing ends (releasing the keyboard back to the player) via Enter -> OnProcess -> SetEditing(false), Esc/CONTROL_CANCEL -> SetEditing(false), or another field grabbing force-process (auto-stops the previous one, frontend.lua:1172-1175). SetEditing(false) calls SetForceProcessTextInput(false, self) (textedit.lua:132) which restores WASD/hotkeys.
