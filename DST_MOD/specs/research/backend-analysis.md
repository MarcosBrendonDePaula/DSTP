## synthesis

- **topPicks:** 
  **1.** 
    - **rank:** 1
    - **title:** Gate the panel control plane (Live Components) behind the panel session
    - **lens:** security
    - **what:** Add `static auth`/per-action session checks to LiveAutomation (saveFlow/deleteFlow/toggleFlow/importFlow/startCapture) and LiveDSTP (sendCommand/broadcast/toggleEventCategory), resolving the dstp_session cookie via PanelAuthStore.hasSession against the action's target serverId, and replace DevAuthProvider in non-dev. Nothing else matters if the control plane is open.
    - **impact:** high

  **2.**

- **themes:** "Three themes dominate. (1) CLOSE THE CENTRAL-HOSTING GAP: the relay's whole purpose is one backend for many internet-reachable hosts, but the code still assumes a trusted LAN — the panel's WebSocket control plane has NO auth (anonymous saveFlow of a script node = full RCE on backend + every DST server), and the open /sync endpoint has an unsanitized path-traversal. The documented 'panel gated by per-server auth' simply isn't enforced where it matters. This is the existential prerequisite for the project's own architecture. (2) MAKE THE LOOP DURABLE AND IDEMPOTENT: nothing in the /dst/sync cycle is persistent or retried — the mod drains events before POSTing, the backend resets the command queue to [] on drain, commands carry no id, and all continuation state (delays, waits, queued events) lives only in per-server worker RAM that the idle-sweep/hang-watchdog/HMR tear down. In parallel, the highest-leverage capability primitives all COMPOUND: a real scheduler unlocks empty-server automation (impossible today), and cooldown + try/catch + a failure-aware log model each prevent a whole class of 'my flow silently spammed / stopped / failed and I can't see why' problems rather than fixing one flow. Favor the moves that ride the existing event-injection + node-registry patterns (scheduler, cooldown, error-branch are backend-only, no Lua) over one-off node additions."

## byLens

**1.** 
  - **lens:** RELIABILITY and correctness — the /dst/sync cycle, FlowEngine error handling, stateful wait execution, worker lifecycle, command/event durability
  - **findings:** 
    **1.** 
      - **title:** Event + command loss on any disconnect/restart: nothing in the sync loop is durable or retried
      - **problem:** The mod drains its event queue with table.remove BEFORE the POST (http.lua DoPoll loops table.remove(state.event_queue,1)) and on a failed/empty response only increments state.connection_errors — the drained events are gone (http.lua:42-45 vs 93-99). Symmetrically the backend command queue is an in-memory Map that handleSync unconditionally resets to [] the moment it builds the response (DSTStateStore.ts:80-84), before the mod confirmed receipt. So backend down for one poll = those game events vanish; backend restart with commands queued = those commands vanish; HTTP response lost in transit = the mod never applied the commands but the backend already cleared them. This is issue #17 but on BOTH sides of the cycle, and there is zero persistence (commandQueues, event_queue, WorkflowInstanceStore are all RAM-only).
      - **proposal:** Make the cycle at-least-once with explicit ack: mod keeps events in the queue and only removes them after a 2xx response; backend stops clearing in handleSync and instead drains up to an acked id the mod echoes back next sync; persist the per-shard command queue in SQLite so a backend restart re-delivers undelivered commands. At minimum document the lossy semantics so authors know bans/purchases can be silently dropped.
      - **impact:** high
      - **effort:** large
      - **evidence:** DST_MOD/scripts/dstp/http.lua:42-45,93-99 (drain-before-POST, no requeue) + DSTStateStore.ts:80-84 (handleSync clears in-memory queue unconditionally)
    **2.** 
      - **title:** No command idempotency/dedup: a retried or duplicated sync re-executes side effects
      - **problem:** Commands are plain {type,data,queued_at} (DSTStateStore.pushCommand:169) with no unique id or monotonic seq, and the mod's ProcessCommands (core.lua:223) applies whatever arrives with no dedup (the only seq in the codebase is the per-frame net_string UI-batch seq, not command-level). Combined with the lack of acks, retrying a sync to recover from loss would double-apply destructive commands: a duplicated kick/ban/spawn_prefab/give_item/heal-max runs twice. The system is forced to choose between loss (today) and duplication.
      - **proposal:** Stamp every queued command with a per-shard monotonic cmd_id. ProcessCommands tracks last_applied_cmd_id per shard and skips ids <= it; the mod echoes acked_cmd_id in its sync payload so the backend drains durably. This unlocks safe retries = effectively exactly-once command application.
      - **impact:** high
      - **effort:** medium
      - **evidence:** DSTStateStore.ts:169 (cmd shape has no id) + DST_MOD/scripts/dstp/core.lua:223 (ProcessCommands applies without dedup)
    **3.** 
      - **title:** Worker lifecycle silently abandons in-flight flow continuations (delays, waits, queued events)
      - **problem:** All flow continuation state lives ONLY in the per-server worker memory: a delay node is a literal setTimeout in the worker (delay/exec.ts:13), pending wait/merge instances live in the worker WorkflowInstanceStore, and not-yet-processed event messages sit in the worker's message queue. Three mechanisms tear the worker down without draining any of it — the idle sweep terminates a core after 10min (sweepIdle:222-231), the hang-watchdog does worker.terminate()+respawn on missed pongs (heartbeat:205-220, comment admits events queued behind the hang are lost), and HMR/restart. So event->delay 5min->ban can lose the ban, and every pending Wait/Merge branch is dropped on respawn. Users see flows that 'sometimes just don't finish'.
      - **proposal:** Expose hasPendingWork() (live setTimeouts + WorkflowInstanceStore.pendingCount) and skip idle-teardown while >0. Persist Wait instances (flowId, waitNodeId, branch contexts, deadline) to the server's SQLite so a respawn rehydrates them (bounded by the existing 1h TTL). Model long delays as a persisted scheduled-resume like wait rather than an in-RAM setTimeout. At minimum log loudly when a core is killed with pending work so loss is observable.
      - **impact:** high
      - **effort:** large
      - **evidence:** delay/exec.ts:13 (setTimeout in worker) + ServerCoreManager.ts:205-231 (heartbeat terminate+respawn and sweepIdle terminate both drop pending) + WorkflowInstanceStore.ts (in-memory Map only)
    **4.** 
      - **title:** Flows run concurrently and un-awaited, causing SQLite read-modify-write races in memory/ai_memory
      - **problem:** In the worker, evaluateEvent calls executeFlow/executeStatefulBranch WITHOUT awaiting (FlowEngine.ts:177-181; evaluateEvent is not async), so back-to-back events plus async nodes (delay/http_request/ai_agent) interleave freely. The memory and ai_memory nodes do non-atomic read-modify-write against SQLite (FlowMemoryRepository.set SELECT-then-INSERT/UPDATE, lines 22-37; memory/exec.ts read->write). Two concurrent runs of the same flow incrementing a counter or appending a list lose updates. The AI conversation-history save (executeAIAgent.ts:261-276 load->append->save) is last-writer-wins: two near-simultaneous chats for one player clobber each other's history.
      - **proposal:** Give the memory nodes an atomic upsert (INSERT ... ON CONFLICT(flowId,key) DO UPDATE) plus an atomic increment/append op. Serialize ai_memory chat-history per (flowId, scopeKey) with a small in-worker async mutex (all runs for one server share the worker thread). Document that flow execution is concurrent so script authors don't assume serial.
      - **impact:** medium
      - **effort:** medium
      - **evidence:** FlowEngine.ts:177-181 (executeFlow not awaited) + FlowMemoryRepository.ts:22-37 (SELECT-then-INSERT/UPDATE) + executeAIAgent.ts:261-276 (load->append->save history)
    **5.** 
      - **title:** /dst/sync is unauthenticated and does unbounded synchronous disk I/O per event
      - **problem:** POST /api/dst/sync has no auth/token/rate-limit (dst.routes.ts:103) and per event does a SQLite insert (EventHistory.create, dst.routes.ts:34-40) plus, when evt.raw is present, a synchronous fs.appendFileSync to data/event_dumps/<type>.jsonl with NO size cap (dst.routes.ts:52-59). The dump write is on the main thread, not the worker, so a flood of events from any host on the network fills the disk and blocks the event loop, taking down ALL servers. EventHistory also has no visible retention.
      - **proposal:** Gate /dst/sync with a shared secret the relay/mod presents (per-install token in relay-config + modinfo), keeping local-network as defense-in-depth not the only line. Cap+rotate the event_dumps files and move the append off the hot path (batch/async, drop over a byte budget). Add a per-shard rate limit (the poll interval is known). Add retention/auto-prune to EventHistory.
      - **impact:** medium
      - **effort:** medium
      - **evidence:** dst.routes.ts:103 (no auth on /sync) + dst.routes.ts:52-59 (unbounded appendFileSync per raw event, sync, main thread)
    **6.** 
      - **title:** foreach over many items can trip the execution-wide loop-guard mid-iteration and abort silently
      - **problem:** The loop-guard counts per-node visits across the ENTIRE execution and aborts at MAX_NODE_VISITS=50 (loop-guard.ts:9,43), while foreach re-runs its each-branch up to MAX_ITEMS=40 (foreach/exec.ts:8,17-22). These are close enough that realistic flows abort mid-loop: two foreach in sequence over a shared downstream node, a foreach whose each-branch node is also reachable by another edge, or a node visited a few times before a 40-item loop, pushes a shared node past 50 visits. On trip the guard logs and returns null, so the loop stops partway with no error surfaced — 'it processed 12 of my 40 players and stopped' looks like data loss.
      - **proposal:** Decouple cycle detection from legitimate iteration: count visits per (nodeId, loop-depth) or reset the per-node counter at each foreach iteration boundary while keeping MAX_TOTAL_STEPS as the runaway backstop; or raise MAX_NODE_VISITS well above MAX_ITEMS. Surface a guard trip into the flow log/capture as a visible error rather than a silent console.error.
      - **impact:** low
      - **effort:** small
      - **evidence:** loop-guard.ts:9,43-50 (per-node MAX_NODE_VISITS=50, execution-wide) vs foreach/exec.ts:8,17-22 (MAX_ITEMS=40, revisits each-branch per item)

**2.** 
  - **lens:** CAPABILITIES — missing flow primitives that unlock big value
  - **summary:** DSTP has a rich node set but its time/control primitives are thin. The ONLY periodic heartbeat is the per-player synthetic `tick` (1Hz, emitted in dst.routes.ts only when players.length > 0) — so there is no way to run a flow on a wall-clock schedule, and nothing fires when the server is empty (restart-at-3am, hourly autosave nag, "boss spawns Sundays"). There is no per-player/per-flow cooldown primitive, so a chat-command flow (`!ia`, `!tp`) can be spammed and every spam runs the full agent/action pipeline (this compounds audit issue #2 where debounce is global, not per-player). There is no error/try-catch branch: a node throw aborts the whole flow (processNode rethrows) with only a console.error — flows cannot recover or notify on failure. There is no sub-flow/call-flow primitive: `list_flows` lets a flow READ others' metadata but nothing can INVOKE another flow, so shared logic (a "send to Discord" or "is-admin gate") is copy-pasted across every flow. The `transform` node covers basic string/number/json ops but has no regex, no date/time math, and no array ops (map/filter/join/slice/sort/sum) — array work forces the RCE `script` node. http_request has no retry/timeout-config/rate-limit and no clean "webhook out" (Discord/Slack) helper. The highest-leverage additions are a real scheduler/cron trigger, a cooldown node, and an error-branch — all three are backend-only (Bun timers in the per-server worker, SQLite for cooldown state), need no Lua, and ride the existing event-injection + registry patterns.
  - **findings:** 
    **1.** 
      - **title:** schedule/cron trigger (time-based entry point, player-independent)
      - **problem:** Flows can only start from a game event or an inbound webhook. The single time source is the synthetic per-player `tick` event built in dst.routes.ts (handleDstSync), and it is gated by `if (Array.isArray(players) && players.length > 0)` — so it fires ~1×/s ONLY while at least one player is online and dies on an empty server. There is no way to express 'every 5 minutes', 'at 03:00', 'every Sunday', or 'every N game-days', and nothing runs when the server is empty (the most common time you WANT automation: nightly restart, off-peak backup, scheduled announcements). A scheduled-restart or 'wipe at season change while nobody's on' flow is impossible today.
      - **proposal:** Add a `schedule` trigger node (triggers/time/schedule/). It is an entry point like `webhook` (no exec, isTrigger), matched in FlowEngine.evaluateEvent by node id against a synthetic `{type:'schedule', scheduleId}` event. Drive it from a single backend timer: in ServerCoreManager (or a new SchedulerService on the main thread, next to the existing `setInterval(checkHealth,15000)` in dst.routes.ts) run a coarse tick (e.g. every 30s) that, per server with enabled flows, computes which schedule nodes are due (cron-ish: interval-ms, or HH:MM daily, or weekday) and calls processAutomationEvent(serverId, {type:'schedule', scheduleId, data:{now}}). Persist last-fire per scheduleId in FlowMemoryRepository so a missed window after a restart doesn't double-fire. Reuse FlowEngine.ensureEventCategories-style scanning to know which servers have schedule flows. Crucially this runs on the MAIN thread independent of /dst/sync, so it fires on empty servers. Expose {{trigger.now}}/{{trigger.scheduleId}}.
      - **impact:** high
      - **effort:** medium
      - **evidence:** frontend/app/server/routes/dst.routes.ts:63-88 (the only periodic source — per-player tick, gated on players.length>0); FlowEngine.ts evaluateEvent:157-185 (entry points are only `trigger`/`webhook`); trigger/ui.tsx:67 lists `tick` as the closest thing to a timer
    **2.** 
      - **title:** cooldown node (per-player / per-key rate limit primitive)
      - **problem:** Nothing lets a flow say 'this player can run this at most once per N seconds'. A chat-command flow (`!ia`, `!tp`, `!home`) fires on every matching chat_message; a player can spam it and each spam runs the full pipeline — for `ai_agent` that's a real LLM call + token cost per spam, for `action` it's repeated kicks/teleports. **4.** 
  - **lens:** SCALE & PERFORMANCE — path to 100 servers / thousands of events/min
  - **summary:** The architecture has a sound isolation story (one worker core per server, idle teardown, hung-core respawn) and per-server SQLite keeps each flow table tiny. But the hot path — every game event — is dominated by repeated full DB reads and full-array WebSocket emits that scale with flow count and event rate, not with what actually changed. The three killers: (1) `findEnabled()` re-reads + JSON-deserializes EVERY flow from SQLite on EVERY event; (2) `syncState()` re-reads all flows + 100 logs and emits them as full arrays over the panel WebSocket on EVERY flow execution; (3) `event_history`/`automation_logs` run a full-scan DELETE-subquery on EVERY insert, and none of the hot tables (flow_memory, automation_logs, event_history) have secondary indexes. Secondary costs: `getServerGroups()` rebuilds + deep-clones the whole player set on every call (called several times per event), and the worker mirror is a `JSON.parse(JSON.stringify(...))` per routed event. At 100 servers the worker model itself (100 OS threads, each with its own bun:sqlite handle + Drizzle + migrate-on-open) becomes the ceiling. None of these are correctness bugs; they're per-event constant-factor and O(n-flows)/O(n-rows) costs that compound under load. The fixes are mostly caching the enabled-flow set, making the panel sync delta/throttled, adding indexes, and switching the per-insert cleanup to periodic.

    **1.** 
      - **title:** findEnabled() reloads + deserializes every flow from SQLite on every event
      - **problem:** FlowEngine.evaluateEvent (FlowEngine.ts:158) calls flowRepo.findEnabled() for EVERY event. At a busy server (chat + tick-per-player-per-second + health deltas) this is dozens to hundreds of events/min, each re-reading and re-parsing every flow the server has. The flow set changes only on save/toggle/delete — it's read thousands of times per write. This is the single hottest repeated cost on the event path and it grows linearly with flow count.
      - **proposal:** Cache the enabled-flow set per server in the worker (the worker owns one server, so a simple `Map<serverId, Flow[]>` or just a module-level array is enough). Invalidate on the existing save/toggle/delete/move paths (they already call invalidateAnalysis — add an invalidateFlows(serverId) alongside, propagated to the worker via a new RPC message like the existing startCapture/stopCapture). collectWatchKeys() and ensureEventCategories already re-scan flows too — share the same cache. Keep the SQLite read only as the cache-miss loader.

      - **effort:** medium
      - **evidence:** frontend/app/server/live/FlowEngine.ts:158 (evaluateEvent → findEnabled per event); FlowRepository.ts:25-30 (full scan + JS filter); schema.ts:5-27 (no index on flows.server_id)

      - **title:** syncState() re-reads all flows + 100 logs and emits full arrays over WebSocket on every flow execution
      - **problem:** Every executeFlow / executeStatefulBranch / executeBranchFromWait ends with this.syncState(serverId) (FlowEngine.ts:503, 614, 681). syncState (FlowEngine.ts:148) does flowRepo.findAll() (full table read + JSON deserialize of every flow) PLUS logRepo.findRecent() (100 rows), then emits the WHOLE flows[] and logs[] arrays as a STATE_DELTA. So every single event that triggers a flow forces a full DB re-read and ships the entire flow list + 100 logs to the panel — even when the panel is closed (the headless/worker host still builds the delta and RPCs it to the main thread). The flat-state-key design exists to avoid exactly this kind of full-array emit, but syncState bypasses it. At thousands of events/min this is the dominant DB read load and the dominant WS payload.
      - **proposal:** Decouple panel sync from flow execution. Only the logs actually change per execution, so: (a) skip syncState entirely when no panel is connected for that server (track subscriber count in LiveAutomation); (b) throttle/coalesce syncState to e.g. once per 1–2s per server (debounced) instead of once per event; (c) emit only the new log row(s) as an append delta, not the full flows[] array (flows didn't change).       - **effort:** medium
      - **evidence:** frontend/app/server/live/FlowEngine.ts:148-153 (syncState reads findAll+findRecent, emits full arrays) and :503,:614,:681 (called per execution)

      - **title:** event_history & automation_logs run a full-scan DELETE subquery on every insert; hot tables have no indexes
      - **problem:** EventHistoryRepository.create (EventHistoryRepository.ts:38) runs `DELETE FROM event_history WHERE id NOT IN (SELECT id ...       - **problem:** DSTStateStore.getServerGroups (DSTStateStore.ts:106) iterates every shard, builds fresh group objects, and spreads/clones every player ({...p, shard_id,...}) into all_players — with NO caching, recomputed from scratch on each call.       - **evidence:** DSTStateStore.ts:106-140 (no cache, clones players); FlowEngine.ts:143-145 (worldFor per flow) + :258-265 (findPlayerInServer); ServerCoreManager.ts:146-161 (groupsFor JSON round-trip) called from route() :167 every event
    **5.** 
      - **title:** One OS worker thread per server with its own SQLite handle + migrate-on-open is the ceiling at 100 servers
      - **problem:** ServerCoreManager spawns a dedicated Worker per active server (ServerCoreManager.ts:99) plus a 2s heartbeat ping to each (HEARTBEAT_INTERVAL_MS). At 100 concurrently-active servers that's 100 OS threads each running a full FlowEngine, each opening its own bun:sqlite connection AND running migrate() on first getDb (connection.ts:79), plus 100 ping/pong message round-trips every 2s on the main thread. Workers are torn down after 10min idle and respawned cold — a server that goes idle then active pays a full worker spawn + DB open + migrate again. The model is great for isolation (a runaway script kills one core) but the fixed per-server overhead (thread stack, sqlite handle, Drizzle instance, migration check) doesn't amortize. The main thread also still does event_history insert + schema autoDetect synchronously per event BEFORE routing to the worker (dst.routes.ts:34-49), so it remains a serialization point.
      - **proposal:** Consider a worker POOL (N workers, servers sharded onto them by hash) instead of 1-per-server once activeCount crosses a threshold — keeps isolation granularity coarser but bounds thread count. Cache the migrate() result (run migrations once per DB file process-wide, skip on subsequent opens — track a Set of migrated paths). Move the per-event event_history insert + autoDetect off the sync hot path (queue + flush, or do it inside the worker) so /dst/sync returns the command response without waiting on DB writes. Make the heartbeat interval scale down (or use a single timer scanning all cores) as core count grows.
      - **impact:** medium
      - **effort:** large
      - **evidence:** ServerCoreManager.ts:99 (Worker per server), :205-220 (per-core 2s heartbeat); connection.ts:79 (migrate on every fresh getDb); dst.routes.ts:34-49 (synchronous event_history + autoDetect per event before routing)
    **6.** 
      - **title:** Secret decryption + vault accessor setup repeated per flow run, with a full FlowRepository.findById on {{env.KEY}}
      - **problem:** installVaultAccessors runs on every flow execution (FlowEngine.ts:439,531) and constructs an EnvironmentRepository + a FlowRepository. The {{env.KEY}} (default-environment) path calls defaultEnvId() which does a full FlowRepository.findById(flowId) — another flows-table read — EVERY time {{env.KEY}} is resolved, on top of the getSecretByEnvId decrypt. scrypt-derived key is cached (good), but a flow that references {{env.X}} in multiple nodes re-reads the flow row and re-decrypts the secret on each reference within the same run (no per-run memo of the resolved value). At high event rates on secret-using flows (e.g. an AI agent firing per chat message) this is repeated DB reads + AES-GCM per node per event.
      - **proposal:** Memoize resolved secrets per run: cache by `environment.<env>.<key>` / `env.<key>` in the per-context sink so the second reference in the same execution returns the already-decrypted (already-masked-tracked) value without a DB hit. Resolve defaultEnvId once per run and stash it on the context instead of re-reading the flow row in the getter. These are pure caching wins with no behavior change.
      - **impact:** low
      - **effort:** small
      - **evidence:** vault-context.ts:49-107 (new repos per install; defaultEnvId does findById on each {{env}} read; no per-run value memo); FlowEngine.ts:439,531 (installVaultAccessors per execution)
    **7.** 
      - **title:** FlowAnalyzer cache and capture buffers are keyed/bounded globally, not per-server — cross-server eviction at scale
      - **problem:** The getAnalysis cache (FlowAnalyzer.ts:15) is a single process-wide Map capped at MAX_CACHE_SIZE=200 with FIFO eviction by flowId. Now that flows run in per-server WORKER processes, each worker has its own module instance, so this is actually per-worker — but the 200 cap means a single server with >200 stateful flows thrashes, and the cache is keyed by flowId only (no server prefix), so it relies on flowId global uniqueness. More importantly getAnalysis is called inside evaluateEvent for every matched trigger; on a cache miss it walks edges backward per wait node. With the findEnabled re-read already happening per event, the analysis lookup is cheap only because of this cache — but the FIFO eviction (not LRU) can evict a hot flow that's used every event while keeping a cold one.
      - **proposal:** Make the analysis cache LRU (touch on get) rather than FIFO-evict-oldest-inserted, so a per-event hot flow is never evicted by a one-off save. Since each worker owns one server, the 200 cap can be per-server-sized or removed (flows are bounded per server anyway). Cheap once the findEnabled cache (finding #1) lands, because the analysis can be precomputed and attached to the cached flow set at load time instead of re-looked-up per event.
      - **impact:** low
      - **effort:** small
      - **evidence:** FlowAnalyzer.ts:15-16,58-68 (single global Map, MAX_CACHE_SIZE=200, FIFO eviction); FlowEngine.ts:176 (getAnalysis per matched trigger per event)
**5.** 
  - **lens:** SECURITY hardening (beyond the accepted RCE-by-design): does the no-auth-on-/dst/sync trust model stay sound now that the relay enables central hosting, and what hardening is worth it without breaking that model.
  - **summary:** The "trusted-LAN, no auth on the data plane" assumption is reasonable for the game→backend POST itself, but two things break it once the relay enables one central backend for many hosts: (1) the panel control plane — flow CRUD, command-push, full state — is NOT actually gated by the panel session at all. LiveAutomation and LiveDSTP declare publicActions but no `static auth`, and the only registered Live auth provider is the demo DevAuthProvider (hardcoded `admin-token`). So any anonymous WebSocket client to the backend can saveFlow/deleteFlow/sendCommand/read all players — the panel password (PanelAuthStore) only guards the REST environments routes, never the Live Components. (2) Several `/api/dst/*` REST endpoints (`/command`, `/dump-mode`, `/players`, `/dumps`, `/record/*`) are wide open, and the event-dump file path is built from unsanitized `evt.type`/`params.type` while serverId is carefully traversal-checked — an inconsistency that is a real path-traversal write/read. Secondary gaps: no brute-force throttle on the real panel login (RateLimiter exists but is only wired to the demo auth.routes), session cookie missing `Secure`, no per-server/global cap on DB-file creation or concurrent flow executions from the unauthenticated sync flood. The vault crypto/masking and serverId charset validation are genuinely solid. None of the proposed fixes touch the deliberate `script`/`call_component` RCE-by-design or the game→sync no-auth choice; they close the gap between the documented model ("panel gated by per-server auth") and what the code enforces.
  - **findings:** 
    **1.** 
      - **title:** Panel Live Components (flow CRUD, command-push, state) have NO auth gate — the password model the docs claim is not enforced on the WebSocket control plane
      - **problem:** CLAUDE.md states 'the admin panel is gated by per-server auth.' That is only true for the REST environments routes (environments.routes.ts checks hasSession). The Live Components that ARE the panel — LiveAutomation (saveFlow/deleteFlow/toggleFlow/importFlow/startCapture) and LiveDSTP (sendCommand/sendPlayerCommand/broadcastCommand/toggleEventCategory) — declare `static publicActions` but NO `static auth`. The contrast is explicit: LiveAdminPanel.ts and LiveProtectedChat.ts DO declare `static auth = { required: true, roles: ['admin'] }`, the two real DSTP panels do not. The only Live auth provider registered (app/server/index.ts:31) is DevAuthProvider, which authenticates hardcoded demo tokens (`admin-token`/`user-token`), unrelated to the panel session cookie. Net: any client that opens the Live WebSocket and sends a saveFlow/sendCommand action runs it unauthenticated — create a flow with a `script`/`call_component` node and you have RCE on the backend and the DST server, with zero password. On a LAN this is masked by network trust; once the relay points many hosts at one central backend (dstp.marcosbrendon.com), the WS endpoint is internet-reachable and this is a full takeover of every connected server.
      - **proposal:** Wire the panel session into the Live auth layer instead of the demo provider. Either (a) add `static auth = { required: true }` + a `static actionAuth` to LiveAutomation/LiveDSTP and register a real LiveAuthProvider that resolves the `dstp_session` cookie via PanelAuthStore.hasSession against the action's target serverId (the serverId is in every action payload / state key), or (b) if the Live framework can't see per-action serverId easily, add an explicit `hasSession(token, serverId)` check at the top of each mutating action (saveFlow, deleteFlow, sendCommand, etc.) reading the cookie from the connection context. Replace/disable DevAuthProvider in non-dev. This is the single highest-value fix and it does NOT change the game→sync trust model.
      - **impact:** high
      - **evidence:** app/server/live/LiveAutomation.ts:79-97 (publicActions, no static auth) and LiveDSTP.ts:32-41 vs app/server/live/LiveAdminPanel.ts:61-72 (static auth present); app/server/index.ts:31 registers only DevAuthProvider; environments.routes.ts:40-51 is the ONLY place hasSession gates a panel surface
      - **effort:** medium
    **2.** 
      - **title:** Unauthenticated /api/dst/* control endpoints can push commands and toggle dump mode to any server
      - **problem:** dst.routes.ts mounts, on the same no-auth `/dst` Elysia as the legitimate game sync: POST /command (pushCommandToServer — arbitrary command type+data to any server_id), POST /dump-mode, POST /record/start|stop, GET /players, GET /servers, GET /dumps, GET /dumps/:type. The mod itself only ever calls /sync; these others exist for the panel/dev but carry no session check. `/command` is a command-injection primitive into the DST server (it queues whatever the relay forwards to the game). With central hosting these are internet-reachable. They duplicate, unauthenticated, capabilities that the (also-unauthenticated, see above) LiveDSTP exposes.

    **3.** 
      - **title:** Path traversal in event-dump file path: evt.type / params.type joined into a filesystem path unsanitized (serverId is checked, these are not)
      - **problem:** On /dst/sync, when an event carries `raw`, the handler does `fs.appendFileSync(path.join(dumpDir, ` + '`${evt.type}.jsonl`' + `))` with `evt.type` taken verbatim from the POST body (dst.routes.ts:58). GET /dumps/:type reads `path.join(..., ` + '`${params.type}.jsonl`' + `)` from the URL (dst.routes.ts:220). Both are attacker-controllable (the sync endpoint has no auth; an event type like `../../config/foo` or `..\..\.env` escapes data/event_dumps). This is the exact threat that getDb so carefully defends for serverId (connection.ts:50 SAFE_SERVER_ID regex + assertSafeServerId), but the same discipline was not applied here — an inconsistency that turns dump-mode into an arbitrary-file-append (write) and arbitrary-.jsonl-read primitive.

      - **title:** Unauthenticated /dst/sync can create unbounded per-server SQLite DBs and accumulate timers — no server_id allow-list / resource cap
